2. Profiles、shape 與 runtime model
====================================================================


2.1 Profiles
- 預設 profile 為 PROFILE_RUNTIME。
  - runtime CFG 可改變 CODE_N、CODE_C、N_LAYERS、D_MODEL、D_FFN，以及選配的 D_LPE
  - N_HEADS 固定為 8
  - 必須檢查 cfg_d_model % 8
  - compile-time 的 MAX_* 限制仍然存在；runtime CFG 必須落在這些上限內
- 允許 PROFILE_FIXED 供快速 bring-up 與 trace alignment 使用。
  - CFG 流程仍走 Top-FSM
  - 但不在 runtime 重新調整 arrays/resources 的尺寸

2.2 LayerNorm 插入規則
- mid LN index：mid_index = (N_LAYERS // 2) - 1
- 若 mid_index < 0，可在明確邊界條件下略過 mid LN。
- end LN 為強制，必須放在所有 transformer layers 之後、FinalHead 之前。

====================================================================
3. Block 分割與 top-level 架構
====================================================================


3.1 必要 blocks
- Top
  - SRAM 唯一擁有者
  - FSM controller
  - runtime cfg registers
  - memory request arbiter
  - base/len 記錄者
- PreprocEmbedSPE
  - embed + SPE
  - LPE 不在線上計算，而是作為 parameter data 載入
- TransformerLayer
  - 單一硬體 layer，透過 time-multiplex 重複使用以支援 N_LAYERS
  - 必須涵蓋 Q/K/V projection、multi-head attention、concat、output projection、residual、FFN，以及演算法流程要求的 layer-local norm 行為
- LayerNormBlock
  - 獨立的 LN block
  - 由 Top 用於 mid LN 與 end LN 插入
- FinalHead
  - transformer 後端 head
  - 包含 final embedding scalar 產生，以及依本規格產生 logits / x_pred

3.2 Top 的排程所有權
- 子 blocks 不得自行配置 SRAM。
- Top 將 base/len/region 等資訊傳給 blocks。
- Top 負責 X_WORK / SCRATCH 的 lifetime 與 free-point、overlap scheduling，以及 output sequencing。

====================================================================
4. 外部介面與協定
====================================================================


4.1 Channel 定義
- ctrl_cmd  : ac_channel<ac_int<16,false>>
- ctrl_rsp  : ac_channel<ac_int<16,false>>
- data_in   : ac_channel<ac_int<32,false>>
- data_out  : ac_channel<ac_int<32,false>>

4.2 「小控制、大資料走 data_in」
- ctrl_cmd 僅承載 opcode（加上保留位元）。
- outmode、addr_word、len_words、param_base_word、debug packed argument 等命令參數，都透過 data_in 以 u32 words 傳送。

4.3 Top modes / receive states
- IDLE
- CFG_RX
- PARAM_RX
- INFER_RX
- HALTED
- 在 overlap mode 下，當 CTX_ACTIVE 正在 compute 時，Top 可接受 CTX_NEXT 的 INFER payload；但任一時刻仍只允許一條 RX payload stream 處於 active 狀態。

4.3.1 Input Elastic FIFO
- Top 內必須有一個 input elastic FIFO（建議名稱：in_fifo）。
- data_in 的 words 先被吸收到 in_fifo，再由 CFG_RX / PARAM_RX / INFER_RX 邏輯消費。
- in_fifo 不是完整 payload buffer，只是小型 decoupling FIFO。
- FIFO 深度由 HLS/tool 設定決定，並必須記錄於 milestone report 中。
- Length check 仍以 receiver 實際消費的字數為準，不以 in_fifo 中暫存的字數冒充完成。

4.3.2 Legacy receive states（僅相容用途）
- 若專案需要 backward compatibility，可額外保留：
  - BIAS_RX：data_in 只允許 bias words（全 u32 raw）
  - WEIGHT_RX：data_in 只允許 weight words（FP32 raw + BITPACK raw）
- 這兩個 states 僅屬 legacy split LOAD_BIAS / LOAD_W 流程。
- 在主路徑 unified PARAM 架構下，正式 receive state 仍以 PARAM_RX 為唯一預設。

4.4 ctrl_cmd 格式
- [7:0]   opcode
- [15:8]  reserved

4.5 ctrl_rsp 格式
- [3:0]   rsp_kind : 0=OK, 1=DONE, 2=ERR
- [7:4]   reserved
- [15:8]  payload : OK/DONE 時為 opcode，ERR 時為 err_code

4.6 Opcode 集合
- 0x01 CFG_BEGIN
- 0x02 CFG_COMMIT
- 0x04 LOAD_W          （主路徑：unified PARAM stream）
- 0x05 SET_OUTMODE     （arg0 走 data_in）
- 0x06 INFER
- 0x07 READ_MEM        （arg0=addr_word, arg1=len_words）
- 0x08 DEBUG_CFG       （arg0=packed debug config）
- 0x09 SET_W_BASE      （arg0=param_base_word）
- 0x03 LOAD_BIAS       [僅 legacy compatibility]
- 0x7F SOFT_RESET

4.6.1 DEBUG_CFG packed argument（arg0 layout，u32）
- [0]     dbg_en
  - 1=啟用 debug checkpoint 條件
  - 0=關閉
- [1]     dbg_halt_en
  - 1=命中 checkpoint 時就地 halt（D1）
  - 0=命中時不 halt
- [3:2]   dbg_action
  - 0=SET/ARM
  - 1=RESUME
  - 2=CLEAR
  - 3=reserved
- [11:4]  dbg_kind
- [19:12] dbg_layer_id
- [27:20] dbg_head_id
- [31:28] reserved
- dbg_kind 的建議固定編碼如下：
  - 1 = X_IN
  - 2 = K_CACHE
  - 3 = V_CACHE
  - 4 = ATTN_OUT
  - 5 = X_OUT
  - 6 = MID_LN_OUT
  - 7 = END_LN_OUT
- DEBUG_CFG(SET/ARM) 只設定斷點條件；不會直接觸發輸出或記憶體搬移。

4.7 OUTMODE
- 0 = OUTMODE_XPRED
- 1 = OUTMODE_LOGITS
- 2 = OUTMODE_NONE
- OUTMODE 控制正常 post-infer 的輸出行為。
- 推論完成後，正常輸出必須自動發生，不需要額外的「output command」。

4.8 Response 順序
- 命令一旦被接受，Top 先回 RSP_OK(opcode)。
- 需要等待 payload 完成或 compute 完成的命令，之後再回 RSP_DONE(opcode)。
- 發生錯誤則回 RSP_ERR(err_code)。
- ERR_DBG_HALT 是特例：Top 會停在 HALTED，而不是回到 IDLE。

4.8.1 READ_MEM 行為
- READ_MEM 的 arg0 為 addr_word，arg1 為 len_words，兩者皆以 u32 word 為單位。
- READ_MEM 只允許在 IDLE 或 HALTED 下進入。
- 若在 CFG_RX / PARAM_RX / INFER_RX、任何 active compute 期間，或任何正常 infer output 尚未結束的期間收到 READ_MEM，必須回 RSP_ERR(ERR_BAD_STATE)。
- 非法 READ_MEM 不得吐出任何 readback payload；其目的在於避免 READ_MEM 的 data_out payload 與正常 infer output 混流。
- Top 在合法狀態收到 READ_MEM 後，必須依序把 len_words 個 u32 words 從 SRAM 送到 data_out。
- 正常 READ_MEM 流程完成後，再回 RSP_DONE(READ_MEM)。
- READ_MEM 屬於 debug / verification / readback 指令，不得被定義為正常 infer output 的必要步驟。
- IDLE 下的 READ_MEM 只用於靜態 readback / verification / bring-up，例如 LOAD_W 完成並回到 IDLE 後的 readback 驗證、known pattern 回讀，或 inference 完成且回到 IDLE 後的靜態 readback。
- HALTED 下的 READ_MEM 只用於 checkpoint halt 之後的 debug readback；TB 必須先接收 halt/meta，再發 READ_MEM。

4.9 主要 error codes
- 0x01 ERR_BUSY
- 0x02 ERR_BAD_STATE
- 0x03 ERR_CFG_LEN_MISMATCH
- 0x04 ERR_BIAS_LEN_MISMATCH [legacy split path]
- 0x05 ERR_W_LEN_MISMATCH
- 0x06 ERR_BITPACK_PAD
- 0x07 ERR_BAD_OUTMODE
- 0x08 ERR_MEM_RANGE
- 0x09 ERR_CFG_ILLEGAL
- 0x0A ERR_TIMEOUT_INTERNAL（選配）
- 0x0B ERR_DBG_HALT
- 0x0C ERR_PARAM_LEN_MISMATCH
- 0x0D ERR_PARAM_BASE_ALIGN（選配）
- 0x0E ERR_PARAM_BASE_RANGE

5. 主要 runtime 流程
====================================================================


5.1 主流程
- CFG_BEGIN -> CFG_RX -> CFG_COMMIT
- SET_W_BASE
- LOAD_W（unified PARAM stream）
- SET_OUTMODE
- INFER

5.1.1 INFER 的正式邏輯主鏈
- input_y 的預設輸入順序為 variable-major；第 k 筆輸入對應 y[var_idx = k]。
- 本版主幹 activation 的正式落點為單一 X_WORK。
- 「active / next」若仍出現在討論或 trace 語境中，僅表示 phase / producer-consumer 語意，不再要求實體雙頁。
- 一般 N-layer 版主鏈如下：
  y
  -> node_feature
  -> preproc_x（映射到 X_WORK）
  -> [TransformerLayer_0 ... TransformerLayer_{N_LAYERS-1}]（以 phase-safe overwrite 共用 X_WORK）
  -> end LN
  -> FinalHead Pass A
  -> SCRATCH.FINAL_SCALAR_BUF
  -> FinalHead Pass B
  -> data_out
- 若 mid LN 啟用，則於 layer = mid_index 後插入 Mid LN，再進入後續 layers。
- 2-layer 範例可寫為：
  y
  -> node_feature
  -> preproc_x（住在 X_WORK）
  -> Layer0_out（映射到 X_WORK；先 KV_MAT，再 token-wise overwrite）
  -> midLN_out（映射到 X_WORK；token-wise in-place）
  -> Layer1_out（映射到 X_WORK；先 KV_MAT，再 token-wise overwrite）
  -> endLN_out（映射到 X_WORK；token-wise in-place）
  -> FinalHead Pass A
  -> SCRATCH.FINAL_SCALAR_BUF
  -> FinalHead Pass B
  -> data_out

5.2 Unified PARAM stream
- PARAM 包含 bias、LayerNorm parameters、所有 linear weights，以及 bit-packed parameters。
- PARAM 透過 param_base_word + offset 定址。
- WeightStreamOrder.h 提供 EXP_LEN_PARAM_WORDS 與 kParamMeta[]，並且它們是 section 順序與 offset 的唯一來源。

5.3 SET_W_BASE 規則
- param_base_word 是 u32 word address。
- 必須滿足 ALIGN_WORDS 對齊。
- 另外也可能要求滿足 W_LANES 對齊。
- 必須落在 W_REGION 範圍內，且保留足夠空間容納 EXP_LEN_PARAM_WORDS。
- 若 range/alignment 失敗，回 ERR_PARAM_BASE_RANGE / ERR_PARAM_BASE_ALIGN。

5.4 LOAD_W 規則
- 必須先成功完成 CFG_COMMIT 與 SET_W_BASE。
- 從 data_in 精確消費 EXP_LEN_PARAM_WORDS 個 u32 words。
- 依序寫入 sram[param_base_word + i]。
- BITPACK sections 必須通過 valid_bits / zero-padding 檢查。
- ternary packed weight sections 必須通過 codebook 檢查：僅允許 00/01/11；若遇 10（illegal）必須視為格式錯誤並回報既有格式錯誤碼。
- 成功後回 RSP_DONE(LOAD_W)。

5.5 Legacy compatibility path
- split LOAD_BIAS / LOAD_W 流程只保留做 backward compatibility。
- 它不是預設架構路徑。
- 若實作 legacy split path，則必須遵守：
  - LOAD_BIAS 只允許在 IDLE 下進入，並切到 BIAS_RX
  - BIAS_RX 期間 data_in 只允許 bias words（u32 raw）
  - LOAD_W 在 legacy path 下切到 WEIGHT_RX
  - WEIGHT_RX 期間 data_in 只允許 weight words（FP32 raw + BITPACK raw）
  - Top 對 BIAS_RX / WEIGHT_RX 都必須逐段驗證 expected_len；少送、多送或超長都要回對應 ERR_*_LEN_MISMATCH
  - legacy split path 的回應順序仍為：先 RSP_OK(opcode)，payload 消費完成後再 RSP_DONE(opcode)

6. 資料表示法
====================================================================


6.1 共通 stream 單位
- DATA_W = 32
- 所有 streams 都以 u32 raw words 傳輸

6.2 FP32 payload 規則
- TB 先把 float/double 來源值轉成 fp32。
- IEEE754 bit pattern 以 u32 raw word 形式送出/儲存。
- 設計端僅在實際使用時，才把 raw bits 轉成 ac_ieee_float<binary32>。

6.3 BITPACK payload 規則
- bit_idx=0 對應 word[0].bit0（u32 內部採 LSB-first）。
- 最後一個 word 中未使用的尾端 bits 必須為 0。
- valid_bits 只對 BITPACK sections 有意義。

6.3.1 Ternary weight SRAM encoding（v12.1 freeze）
- ternary codebook 固定為：00=0、01=+1、11=-1、10=reserved/illegal。
- 每個 ternary weight 佔 2 bits；每個 32-bit word 固定打包 16 個 ternary weights。
- packing 採 LSB-first：第 i 個權重放在 word[(2*i)+1 : 2*i]；例如第 0 個權重對應 word[1:0]，第 15 個權重對應 word[31:30]。
- 每個 ternary 權重矩陣的最後一個 payload word 都必須帶有 last_word_valid_count，且合法範圍固定為 1..16；若最後一個 word 滿載，last_word_valid_count=16。
- tail padding bits 必須為 0；亦即最後一個 word 中超出 last_word_valid_count 的高位 pair 一律為 0。
- 每個 ternary 權重矩陣在主流程中必須有 per-matrix inv_s_w；s_w 保留數學語意，但不是主流程必備儲存欄位。
- matrix record 在本版定義為邏輯 section format，至少包含：matrix_id、rows、cols、num_weights、payload_words、last_word_valid_count、inv_s_w、layout_kind、payload。
- 上述 matrix record 只定義邏輯格式，不代表在本版手動固定最終 physical SRAM base offsets；實體 offsets 仍由 SramMap / WeightStreamOrder / offline generator 決定。
- 權重矩陣的數學表示維持 W[out][in]（out=Row, in=Colume）；「in contiguous」僅表示實體線性化/存放順序，不改變數學表示。

6.4 H matrix 的正式數學定義
- H 的唯一數學定義為 H[c][v]。
- 列(Row) = check index c。
- 行(Colume) = variable index v。
- 後續所有 H 的敘述，若未特別註明為 physical layout，皆必須回到此數學定義。

6.4.1 H 的 logical definition 與 physical layout 必須分開描述
- H[c][v] 是數學定義，不等於 SRAM 內部的實體布局。
- physical layout 可使用 H_BY_VAR、H_BY_VAR_BITPACK、H_VAR_TO_CHECK_ADJ、adjacency-by-variable 等名稱描述。
- 若為支援 variable-major input_y streaming 與 check_parity 更新，reference implementation 應優先採用 H_by_var 或等價 adjacency-by-variable 方向。

6.5 Runtime access 術語
- row-scan：固定某個 check c，掃所有 variable v。
- column-access：固定某個 variable v，取得所有 check c 的連接資訊。
- 若 input_y 採 variable-major stream，check_parity 更新應以 column-access 或 adjacency-style fetch 為主，不應把 full row-scan 當作預設主路徑。

6.6 input_y 的預設 stream order
- input_y 預設採 variable-major stream。
- 第 k 筆輸入對應 y[var_idx = k]。
- 更精確地，對 variable-side ingest，單個 y[v] 輸入後，Top 應緊接提供該 variable 對應的 embedding / parameter window（或等價固定順序資料），以利順序管線；不得把「先收完整 input_y、再逐筆補 variable-side embedding」當成本版正式主順序。
- check-side Stage B 仍以「全部 input_y 處理完成」為前提；check_feature / check-side embedding 不因 variable-side 順序管線而提前越界。
- 若未來允許其他輸入順序，必須由 CFG 或 metadata 明示；不得只用口語描述模糊帶過。

6.7 Preproc / ingest 名詞定義
- check_parity[c]：Preproc ingest 階段的 local accumulator。
- check_feature[c]：全部 input_y 處理完成後，由 check_parity 轉成的 scalar feature；其值域預設為 {+1, -1}。
- check token：Preproc 把 check_feature 與 embedding / positional information 結合後形成的完整 token 向量。
- node_feature：由 variable-side feature 與 check-side feature 共同組成的 Preproc 輸入集合。

6.8 base mask 與 runtime derived rule 語意
- base mask 的 bit 語意固定為：1 = connected，0 = not connected。
- 參數流 / SRAM 中只保存一張 base source mask；ring1 / ring2 不作為獨立 persistent tables 存在。
- ring1 / ring2 屬於 runtime derived policy：由 base mask、token type comparator 與 head-group comparator 即時計算。
- attention datapath 內部應先維持 connected / not-connected 語意，再由規則層轉為 allow / block 判定，避免在程式中出現未經文件定義的語意反轉。

7. 記憶體架構
====================================================================


7.1 實體記憶體假設
- 目前唯一有效的主架構是：single main SRAM + deterministic 1-issue arbitration。
- 每個 cycle 最多只能有一次 memory access（讀或寫擇一）。
- 舊版 dual-port 的敘述只保留為 legacy，不得覆蓋本規則。

7.2 Logical regions
- W_REGION
  - 存放 weights / params / bitpack
  - 對 ternary 權重矩陣至少包含 packed payload、per-matrix inv_s_w 與 matrix record metadata
  - 受 SET_W_BASE + WeightStreamOrder.h 管控
- IO_REGION
  - 選配的 input_y staging / output staging / debug dump / metadata tables
- X_WORK
  - 放置單一 token × d 主工作矩陣
  - shape 語意為 x[token][d]；token 是列(Row)，d 是行(Colume)
- SCRATCH
  - 暫存區
  - 包含 FinalHead scalar buffer 的強制落點，以及 S1 KV cache 的放置區
- 若有其他 legacy/debug persistent regions，必須在 SramMap.h 中明確宣告

7.3 X_WORK 架構
- X_WORK 的正式預設配置為單一實體主工作區，而非實體雙頁 baseline。
- X_WORK_WORDS = align_up(T_TOKENS * D_MODEL, ALIGN_WORDS)。
- 實體上只保留一塊 token × d 工作區；不得要求 page-to-page memcpy。
- 「active / next」若在文件、trace 或 code comment 中出現，只作為 phase 名稱：
  - active：當前被 consumer 視為舊值來源的邏輯輸入
  - next：當前 producer 正在形成的新值語意
- 不得把 active / next 語意誤解為必須配置兩個實體主工作頁或額外 page bank。

7.4 X_WORK overwrite 規則與 final-layer no-writeback
- 本版採 single-X_WORK 的 phase-safe overwrite；正確性以 liveness / producer-consumer / free-point 為準，不得依賴舊式雙頁切換。
- 對 layers 0 .. N_LAYERS-2：
  - 先完成該 layer 的 KV_MAT：由 X_WORK 產生 K/V，並寫入 SCR_K / SCR_V
  - Phase B 對每個 q token：
    - 讀 X_WORK[q_token][:] 作為 Q / residual source
    - compute
    - 允許把結果覆蓋回同一個 X_WORK[q_token][:]
- 某 token 在 X_WORK 上的舊值，只有到達 free-point 後才允許覆蓋；其 free-point 必須滿足：
  - K/V build 所需讀取
  - 該 token 的 Q / residual source 所需讀取
  才允許覆蓋。
- 對 layer N_LAYERS-1：
  - 仍不得要求額外的第二份主工作區 full writeback
  - final output 保留在 X_WORK，供 End LN / FinalHead Pass A 消費

7.5 Memory request abstraction
- Top 以抽象 request 型式仲裁所有記憶體存取：
  {addr_word, is_write, wdata, tag(region, block_id)}

7.6 Deterministic single-port priority
- 固定服務優先序如下：
  1) 必要 writeback
  2) X_in 讀取
  3) W_REGION prefetch
  4) DEBUG / READ_MEM
- 即使在同優先級內做公平化，規則仍必須是 deterministic。

7.7 W_REGION 寬讀語意
- W_LANES 固定為 8。
- 外部 data_in/data_out 仍維持 u32 為基礎。
- 內部可把寬讀行為視為連續 8 次 u32 讀取，再配合 local buffering。
- segment base/len 的對齊與 padding，必須符合 SSOT 宣告的 W_LANES 規則。
- 禁止在 inner-most loop 中做隨機逐字 weight 讀取。

7.8 Local buffering 與 streaming
- 權重必須以 burst/tile 方式抓取，並在 local regs / FIFO / line buffer 中重複利用。
- softmax partial values 不得寫入 SRAM。
- FFN1 hidden 與 attention outputs 應儘量以 streaming 方式傳遞；若必須 spill，只能 spill 到已宣告的 scratch。
- ingest 階段不應要求每筆 y[v] 都做 full row-scan；H 的 layout 與 fetch policy 必須支援 variable-major 下的 column-access 或 adjacency-style fetch。
- 若 base mask 需在 attention 中重複使用，必須定義適當的 tile / local buffer / decode 策略；不宜在 inner-most score loop 中對 SRAM 做零碎隨機 bit 讀取。

7.9 Scratch 策略 S1：KV cache
- K 與 V 必須 materialize 到 SCRATCH。
- Q 保持 on-the-fly，不 materialize。
- 對單一 context：
  - X_WORDS = cfg_n_nodes * cfg_d_model
  - X_WORK_WORDS = align_up(X_WORDS, ALIGN_WORDS)
  - SCR_K：shape = [cfg_n_nodes, cfg_d_model]，base = scratch_base + 0 * X_WORK_WORDS，len = X_WORK_WORDS
  - SCR_V：shape = [cfg_n_nodes, cfg_d_model]，base = scratch_base + 1 * X_WORK_WORDS，len = X_WORK_WORDS
  - SCRATCH_WORDS(S1) = 2 * X_WORK_WORDS
- 一旦整張 K/V 已 materialize 完成，後續 attention token-wise 運算允許依 liveness 規則覆蓋回 X_WORK。
- 小型 softmax partials、單一 node 的 acc vectors 與 sumexp/max 等暫存，應保留在 regs/local buffers，不占用 scratch page。

7.10 IO region staging 規則
- 本版正常 input_y 一律直接串流，不作 SRAM staging。
- 本版正常 output 一律直接經 data_out 串流輸出，不作 SRAM staging。
- IO_REGION 不作為本版正常 input/output 主路徑；若後續要導入特殊 staging / dump / debug 模式，必須另立條款。
- SET_W_BASE 不得指向 IO_REGION。

====================================================================
8. Multi-context overlap
====================================================================


8.1 必要能力
- 至少支援兩個 contexts：CTX0 與 CTX1。

8.2 目標
- 當 CTX_ACTIVE 尚在運算時，Top 可先接受下一筆 CTX_NEXT 的 INFER payload，並在資源允許時先做前段工作。

8.3 約束
- 任一時刻只允許一條 RX payload stream active。
- 若兩個 contexts 都被佔用，INFER 必須回 ERR_BUSY。
- 只要任何 inference 仍在飛行中，CFG/PARAM 類命令都禁止。

8.4 Per-context persistent buffers
- 每個 context 都必須擁有：
  - 自己的單一 X_WORK
  - 自己的 SCRATCH live 子區；若採用 S1 KV cache，則有自己的 SCR_K / SCR_V
- 這些 buffers 屬於 CTX_PERSIST；不同 live contexts 的 X_WORK 與 SCRATCH live 區域不得 alias。
- 若未來另做 optional dual-page / tile ping-pong 微架構，必須明列為選配實作，不得回頭改寫本版預設 contract。

8.5 Overlap 模式下的輸出順序
- data_out 的正常輸出，必須永遠按照 INFER 命令被接受的順序。
- 外部介面不得新增 ctx_id。

====================================================================
9. Transformer layer 執行模型
====================================================================


9.1 PreprocEmbedSPE / ingest
- input_y 預設可直接串流進入 Preproc，不必先落主 SRAM。
- 對 variable-side token：
  - y[v] 到來時可直接計算 abs(y[v]) 作為 var_feature[v]。
  - sign(y[v]) 形成 hard_bit[v]。
  - 同一個 y[v] 輸入後，Top 應緊接送入該 variable 對應的 embedding / parameter window（或等價固定順序資料），以維持順序資料流。
- check_parity[c] 屬於 local state；預設以 local regs / local array 保存，不應被定義成每筆 y 都需對 SRAM 做 read-modify-write 的 rolling state。
- 每個 y[v] 到來時，依 hard_bit(y[v]) 與 H_by_var[v]（或等價 adjacency-by-variable 讀法）更新 check_parity[c]。
- 全部 input_y 處理完成後，必須執行 check-side Stage B：
  - check_parity[c] -> check_feature[c] ∈ {+1, -1}
  - 再由 check_feature[c] 進入 check-side Preproc / embedding 路徑
- node_feature 由 variable-side 與 check-side feature 共同形成。
- Preproc 本體結合 embedding 與 SPE。
- LPE 不在線上計算，而是預先載入的 parameter data；目前正式主規格只使用 lpe_token，不把未使用的 lpe 與 lpe_proj.{weight,bias} 視為主路徑必要部分。
- LPE layout：
  - cfg_d_embed = cfg_d_model - cfg_d_lpe
  - LPE_BASE = cfg_d_embed
  - x[token][LPE_BASE .. cfg_d_model-1] 對應 lpe_token[token][0 .. cfg_d_lpe-1]
- flatten 規則為 token-major、lpe-dim-minor。

9.2 採用 S1 KV cache 的 attention
- Phase A：由 X_WORK 產生 K 與 V，並 materialize 到 SCR_K / SCR_V。
- 在 Phase A 完成前，不得覆蓋任何仍需作為 K/V source 的舊 X_WORK。
- Phase B：對每個 q token
  - on-the-fly 產生 Q
  - 讀取 base mask，並以 runtime comparators 派生 ring1 / ring2 / allow-block 規則
  - 以 single-pass online softmax 掃描 K/V，於同一輪掃描中更新 max / normalization state / weighted accumulation
  - 形成各 head 的 context 後，立即做 Wo partial accumulation
  - output projection
  - residual
  - 把 attention sublayer 結果寫回 X_WORK[q_token][:]（token-safe overwrite）
- head-group 規則：
  - 當 N_HEADS = 8 時，head 0..3 視為前半 group，head 4..7 視為後半 group。
  - 哪一個 group 使用哪一條 rule（例如 rule1 / rule2），必須由 SSOT 或 generator 明確給定；不得在程式中以隱含慣例假設。
  - 本版對應 algorithm_ref 的凍結選擇為：
    - head 0..3 -> group0 -> rule1 -> one_ring_mask
    - head 4..7 -> group1 -> rule2 -> second_ring_mask
  - 若未來支援非 8-head 版本，head-group 切分規則同樣必須由 SSOT 明示。
- Top-managed active chain 的 Phase B 封包 / window 語意規則：
  - Phase A / Phase C 若採固定 deterministic 順序，且 payload 語意可由 phase_id + token/tile 範圍 + sequence order 唯一推出，則可不額外擴充欄位。
  - Phase B 因同時涉及 Q-source、Wq、K/V scan、mask、Wo 與 output handoff，不得只靠隱含 sequence 慣例解碼。
  - 對 Top-managed Phase B，至少需能顯式區分：
    - head_group_id
    - subphase_id（或等價且可由 SSOT 唯一反推的欄位）
- ring1 / ring2 只能作為 runtime derived policy / local temporary，不得在 X_WORK / SCRATCH / W_REGION 中宣告 persistent storage。
- Phase C：FFN 逐 node 套用在 X_WORK 上，不需要額外的主工作頁。
  - 讀一個 node 進 regs
  - 計算 FFN
  - residual
  - 寫回同一個 X_WORK node
- 本版在 layer 內對 X_WORK 的正式主結果寫回，原則上以 residual 完成後為準；未加 residual 的中間值不作 X_WORK 正式寫回。
- 惟 K/V build 這類為後續 attention 提供的必要中介資料，仍應依既定 Phase A 寫入 SCR_K / SCR_V。

9.3 Residual、LayerNorm 與 stage boundary 規則
- Residual 不得依賴模糊不清的同 cycle bank conflict 行為。
- LayerNormBlock 為獨立 block；Mid LN 為 Top 額外插入的 LayerNormBlock，End LN 為所有 transformer layers 之後、FinalHead 之前的強制節點。
- LayerNorm 必須採 two-pass token-wise 風格：
  - pass1 只讀，用於求 mean/var
  - pass2 再讀同一 token 的 X_WORK，並寫回同一個 X_WORK token 槽位
- 禁止在同一 pass 中對同一 token slice 同時讀寫。
- 「in-place」只允許發生在 pass 邊界之後，不得省略 pass1 / pass2 的資料依賴。
- FinalHead Pass A 與 Pass B 的分界如下：
  - Pass A：FinalEmbedding 產生 s_t，並寫入 FINAL_SCALAR_BUF
  - Pass B：OUT_FC 對 FINAL_SCALAR_BUF 做 reduction，輸出 logits / x_pred

9.4 Softmax approximation（single-pass online）
- 自 v12.1 起，attention 的正式 softmax 規則為 single-pass online softmax；two-pass（pass1 找 max、pass2 累積 sumexp/acc_vec）不再是預設主規格。
- 使用 LUT exp + reciprocal 風格。
- 禁止 std::exp 與直接除法。
- 唯一 SSOT 為 SoftmaxApprox.h（以及與其一致生成的 SoftmaxApproxLutData.h）。
- 必須集中定義：
  - SOFTMAX_NEG_T
  - SOFTMAX_EXP_LUT_SIZE
  - SOFTMAX_RCP_LUT_SIZE
  - SOFTMAX_EPS
  - SOFTMAX_EXP_IDX_SCALE
  - SOFTMAX_RCP_IDX_SCALE
  - SOFTMAX_SUMEXP_MAX
  - exp_lut / rcp_lut 內容
- online 狀態至少包含：
  - m：目前為止的 row max
  - l：對應 m 的 normalization sum
  - acc：weighted value accumulation（可為 tile 版本）
- 必須明確定義數值穩定化策略：
  - 當新 score 高於目前 m 時，需依 online softmax 規則同步重縮放 l 與 acc
  - exp / reciprocal LUT 仍為唯一近似來源，不得在 datapath 內自行做 ad-hoc division
- 固定 index mapping 規則如下：
  - x = score - ref_max，先做 x_clamp = clamp(x, -SOFTMAX_NEG_T, 0)
  - exp_lut index：idx = sat_u((-x_clamp) * SOFTMAX_EXP_IDX_SCALE)
    - SOFTMAX_EXP_IDX_SCALE = (SOFTMAX_EXP_LUT_SIZE - 1) / SOFTMAX_NEG_T
    - 該比例常數必須在 SSOT 中預先計算，不得在 datapath 內用除法即時計算
    - x=0 對應 idx=0；x=-SOFTMAX_NEG_T 對應 idx=LAST
  - reciprocal path：sumexp_safe = clamp(sumexp, 1, SOFTMAX_SUMEXP_MAX)
  - rcp_lut index：idx = sat_u((sumexp_safe - 1) * SOFTMAX_RCP_IDX_SCALE)
    - SOFTMAX_RCP_IDX_SCALE = (SOFTMAX_RCP_LUT_SIZE - 1) / (SOFTMAX_SUMEXP_MAX - 1)
    - 該比例常數同樣必須在 SSOT 中預先計算
- 建議 bring-up 預設：
  - SOFTMAX_NEG_T = 10（或 12）
  - SOFTMAX_EXP_LUT_SIZE = 256
  - SOFTMAX_RCP_LUT_SIZE = 256
  - SOFTMAX_SUMEXP_MAX = 256（需與 LUT 生成參數一致）
  - SOFTMAX_EPS = 1e-8（fp32）或 fixed-point 的 1 LSB

====================================================================
10. FinalHead 架構
====================================================================


10.1 目前唯一有效的 FinalHead 行為
- FinalHead 不得使用 X_WORK 或任何未宣告的 next-stage 暫存區作為 readout staging。
- final-layer no-writeback 規則持續有效。
- End LN 是 FinalHead 的唯一前置強制節點。
- FinalHead 的 token-wise scalar s_t 必須存到 SCRATCH.FINAL_SCALAR_BUF，不得放在 X_WORK 或其他未宣告 staging 區。
- logits 必須以 direct streaming output 方式產生，而不是強制先落地到 SRAM。

10.2 Two-pass readout
- Pass A：
  - 對每個 token t，計算 s_t = FinalEmbedding(LN(x_last[t][0:D_MODEL]))
  - 把 s_t 存入 SCRATCH 中的 FINAL_SCALAR_BUF[t]
- Pass B：
  - logits[c] = 對所有 t 做 OUT_FC[c,t] * FINAL_SCALAR_BUF[t] 的加總
  - 對 c 採 class tiling（例如 TILE_C=8 或 16）
  - 一個 tile 的累加放在 regs 中完成
  - tile 完成後，若 OUTMODE_LOGITS 被選中，就立刻把該 tile 串流輸出到 data_out
  - 禁止在未宣告的主工作區別名中配置額外的 OUT_LOGITS_BUF

10.3 輸出政策
- OUTMODE_LOGITS：
  - 依序串流輸出 OUT_DIM 個 logits
- OUTMODE_XPRED：
  - 依既有 sign / logits*y 規則由 logits 推出 x_pred
  - 依模型定義輸出對應 bits/words
- OUTMODE_NONE：
  - 抑制正常 infer output，保留 data_out 給 debug / meta / readback 流程使用

10.4 X_WORK / staging 禁止用途
- 不得把 X_WORK 的尾端切割、別名化或宣稱為：
  - FINAL_SCALAR_BUF
  - OUT_LOGITS_BUF
  - 任何 FinalHead readout staging
- FinalHead readout staging 若需要 SRAM，只能落在 SCRATCH 或明確宣告的其他合法 region。
- IO staging 仍只能使用 IO_REGION，不得反向侵入 X_WORK。

10.5 Scratch liveness
- FINAL_SCALAR_BUF 的 lifetime 必須延續到 Pass B 完成為止。
- Scratch 配置時，必須確保它不會與 FinalHead 仍需使用的其他 live data 重疊。

====================================================================
11. Debug 與 READ_MEM 政策
====================================================================


11.1 READ_MEM 的地位
- READ_MEM 僅供 debug / verification / readback 使用。
- 正常推論輸出不得依賴 READ_MEM。
- READ_MEM 只允許在 IDLE 或 HALTED 下執行。
- IDLE 下的合法用途只限於靜態 readback / verification / bring-up，例如 LOAD_W 完成並回到 IDLE 後的 readback 驗證、known pattern 回讀，或 inference 完成且回到 IDLE 後的靜態 readback。
- 在 CFG_RX / PARAM_RX / INFER_RX、任何 active compute 期間，或任何正常 infer output 尚未結束的期間收到 READ_MEM，必須回 RSP_ERR(ERR_BAD_STATE)，且不得吐出任何 readback payload。
- 上述限制的目的，是避免 READ_MEM 的 data_out payload 與正常 infer output 混流。

11.2 D1 halt-in-place debug 策略
- DEBUG_CFG 用來配置 checkpoint 條件。
- 在 INFER 期間命中 checkpoint 時：
  1) Top 進入 HALTED
  2) 所有後續 compute 與 SRAM writes 立即停止
  3) ctrl_rsp 回 RSP_ERR(ERR_DBG_HALT)
  4) data_out 依序送出 meta0=addr_word，再送 meta1=len_words
  5) TB 再發 READ_MEM(meta0, meta1) 做 readback
  6) 完成 readback 後，才允許由 DEBUG_CFG(RESUME) 繼續同一次 inference，或由 SOFT_RESET 中止

11.3 必要順序
- ctrl_rsp 的 ERR_DBG_HALT 必須先於 meta0/meta1 產生。
- Top 不得因為 data_out backpressure 而延後 ctrl_rsp。
- 因此內部必須有 2-word 的 meta buffer。
- HALTED 下的 READ_MEM 必須發生在 meta0/meta1 送出之後。
- 非法 READ_MEM 只回 RSP_ERR(ERR_BAD_STATE)，不得再額外吐出 readback payload。

11.4 HALTED 狀態允許的命令
- READ_MEM
- DEBUG_CFG(RESUME/CLEAR)
- SOFT_RESET

11.5 建議的 checkpoint 種類
- X_IN
- K_CACHE
- V_CACHE
- ATTN_OUT
- X_OUT
- MID_LN_OUT
- END_LN_OUT

11.6 D0 legacy 路徑（非預設）
- 若未來需要「不中斷推論、但仍保留 checkpoint snapshot」，可選擇 D0 copy-to-PERSIST_DEBUG 路徑。
- 啟用 D0 時，SramMap.h 必須額外宣告 PERSIST_DEBUG 類區域。
- 命中點需把指定資料 copy 到該固定區域，並在流程回到 IDLE 後再由 TB 以 READ_MEM 讀回。
- 目前 v12 的預設 debug 策略仍是 D1 halt-in-place；D0 只保留為 legacy/可選擴充。

====================================================================
12. 量化架構
====================================================================


12.1 v1 正式凍結範圍
- 本版先只實作 AECCT 原生 linear quant；不實作額外 grouping / zone 分區量化。
- 原生 linear quant 僅適用於下列線性權重與其 activation 邊界：
  - Wq
  - Wk
  - Wv
  - Wo
  - Wff1
  - Wff2
- 上述線性權重採 ternary {+1, 0, -1}。
- 上述線性運算的 activation 邊界採 INT8。
- 其他路徑在 v1 一律維持 FP32：
  - embed
  - LPE token
  - LayerNorm 前後中間值
  - softmax 與其 online state 的數值語意
  - residual / add / concat 等非原生 linear path
  - FinalEmbedding（s_t）
  - FinalHead / OUT_FC / logits / x_pred
- 後續若要把部分非原生 linear path 改成 E4M3 或其他格式，必須以新增條款獨立描述；本版不預設、不暗含。

12.2 固定 s_x / inv_s_x（v1 不做動態更新）
- v1 保留 per-layer / per-slot 的 s_x（以及實作上等價的 inv_s_x）概念，但其值在一次 inference 期間固定。
- Unified PARAM stream 仍保留 QUANT_SX_* 等 quant parameter sections，但在 v1 中它們不是 seed；而是本次/本模型直接使用的固定值。
- v1 不啟用 ScaleEstimator，不做 runtime smoothing、EMA、NR reciprocal 動態更新，也不要求在 inference 後回寫新的 ScaleTable。
- 若實作偏好使用 reciprocal 形式，可在 block 內把固定 s_x 轉成 inv_s_x cache 使用；但其數值語意仍視為「固定配置」，不是動態追蹤。
- 在進入 quantized linear kernel 前，activation 的量化必須包含 clamp / sat；不得允許 overflow 或 wrap-around。
- v1 預設採對稱 signed INT8 飽和語意，建議以 [-127, +127] 為 compare/tuning 基準；若 reference/TB 後續統一為其他 signed INT8 可表示範圍，必須於 SSOT 與 TB 一起同步，且不得改成 overflow wrap-around。

12.3 s_w / inv_s_w 規則（v1 正式要求）
- 權重端必須保留 s_w 的量化語意。
- 但 v1 正式規定：外部 parameter stream / WeightStreamOrder 只需保存 inv_s_w = 1 / s_w，不要求獨立保存 s_w。
- 對每個 ternary 權重矩陣，主流程 metadata 的最小粒度為 per-matrix inv_s_w。
- 若 host / notebook / 轉檔流程原始持有的是 s_w，則必須在離線生成或 streamer 準備階段先轉成 inv_s_w 再送入 unified PARAM stream。
- quant parameter 的 SSOT / metadata 必須能區分 activation 端的 s_x 類參數與權重端的 inv_s_w 類參數。
- Quantized linear kernel 內部若需要權重端比例還原，應使用 inv_s_w（或與其數學等價的固定形式），不得在 datapath 內臨時依賴未定義的 s_w 來源。

12.4 v1 的量化邊界與結果 dtype
- 建議的 v1 線性 kernel 邊界如下：
  - FP32 activation -> 依固定 s_x 量化為 INT8（含 sat）
  - ternary weight 與 INT8 activation 做線性運算
  - 依固定 inv_s_w（以及需要時的等價固定比例）完成該 kernel 的比例處理
  - kernel 輸出回到 FP32 domain
- v1 不要求把整條 residual / LN / softmax / FinalHead 主鏈一起改成低精度。
- FINAL_SCALAR_BUF、OUT_FC reduction、logits/x_pred streaming 均維持 FP32 語意。

12.5 保留但不進主流程的項目
- grouping / zone 分區量化
- QuantDesc.h / QuantConfig.h 的正式量化配置表
- SET_QCFG 或等價量化配置命令
- 更廣義的 pre-infer configurable data quantization external contract
- 上述項目在本版只保留 reserved 身分，不得成為主 datapath、主驗證流程或主 contract 的必要前提。

12.6 未來可選方向（非本版義務）
- 後續若要導入 E4M3，優先建議從非原生 linear path 的局部 domain 著手，例如 FinalHead 內部暫存或其他明確邊界的 FP32 island。
- 在未新增正式條款前，不得把 E4M3、grouping / zone、SET_QCFG、或動態 scale table 行為偷偷視為已啟用。

====================================================================
13. Weight / parameter 順序
====================================================================


13.1 一般原則
- Top 不應理解參數內容的語意。
- 順序由 TB-side streamer 決定，但該順序必須與 WeightStreamOrder.h 完全一致。

13.2 Grouped ordering 規則
- H_BY_VAR_BITPACK / H_BY_VAR_ADJ（名稱必須反映 runtime access 方向）
- embed
- LPE token
- BASE_SRC_MASK（僅保存 base mask；ring1 / ring2 為 runtime derived）
- quant parameters（v1：固定 s_x 與必要的 inv_s_w；grouping/zone 類配置僅 reserved，不進主流程）
- 依 type 分組的 attention weights（跨 layers）
  - Wq(layer0..)
  - Wk(layer0..)
  - Wv(layer0..)
  - Wo(layer0..)（若存在）
- 依 type 分組的 FFN weights（跨 layers）
  - Wff1(layer0..)
  - Wff2(layer0..)
- LayerNorm parameters
- 其他 FinalHead parameters
- WeightStreamOrder metadata 應能描述：
  - layout_kind（例如 H_BY_VAR_BITPACK / H_BY_VAR_ADJ）
  - mask_kind（例如 BASE_SRC_MASK）
  - logical_index_order
  - storage_semantics

13.3 Heterogeneous layer sizes
- 若每層的 words 數不同，這些 per-layer word counts 只能來自 WeightStreamOrder.h。
- CFG 不得額外提供第二套獨立的 per-layer word-count 來源。

14. SramMap / offline generator 要求
====================================================================


14.1 必須有 offline generation
- base offsets 與 region sizes 不得手動維護。
- scripts/gen_sram_map.py 或等價工具，必須負責生成 SramMap.h 與相關 descriptors。

14.2 必要 metadata
- tensor/region name
- base offset
- size in words
- dtype
- valid_bits（供 BITPACK tail 使用）
- region
- lifetime_class
- ctx_id（若適用）
- alias_group（當 scratch overlay 被刻意允許時）
- align_words
- pingpong_tag（若適用）
- layout_kind
- logical_shape
- index_semantics
- storage_semantics

14.3 M16 / generated-view 相容性
- 若存在 gen/ 版本 headers，design/TB 必須一致地 include generated 版本。
- bring-up 用的 legacy headers 可以保留參考，但不得成為第二套 SSOT。

====================================================================
15. TB 與驗證流程
====================================================================


15.1 主要 TB 流程
- 依 ModelDesc.h 的精確欄位順序組出 cfg words
- CFG_BEGIN -> 送 cfg -> CFG_COMMIT
- SET_W_BASE
- LOAD_W，送出精確的 PARAM stream
- SET_OUTMODE -> INFER -> 送 input_y
- input_y 的預設 test stimulus 順序應與 spec 一致，採 variable-major。
- 依 OUTMODE 與 ModelShapes.h 的 expected lengths 讀取 data_out
- 與 golden outputs 比對
- 視需求對 x_pred 與 logits 兩種模式都重複驗證

15.2 expected length 來源
- CFG 長度來自 ModelDesc.h
- INFER/OUT 長度來自 ModelShapes.h
- PARAM section 長度與 metadata 來自 WeightStreamOrder.h

15.3 Overlap 驗證
- 若啟用 overlap mode，TB 應做壓力測試，驗證是否能提早接受下一筆 inference。
- 若未做 overlap stress，Known limitations 必須明寫。

15.4 Debug 驗證
- 對 D1，TB 應支援：
  - SET_OUTMODE(OUTMODE_NONE)
  - DEBUG_CFG(SET/ARM)
  - INFER
  - 接收 ERR_DBG_HALT + meta0/meta1
  - READ_MEM(meta0, meta1)
  - DEBUG_CFG(RESUME) 或 SOFT_RESET
- TB 另須驗證：
  - IDLE 下 READ_MEM 合法，例如 LOAD_W 完成並回到 IDLE 後的 readback
  - HALTED 下 READ_MEM 合法，且必須在接收 halt/meta 之後才發出
  - active infer / 正常輸出未結束期間發 READ_MEM，必須回 RSP_ERR(ERR_BAD_STATE) 且不吐 payload

15.5 H layout / ingest 驗證
- TB 必須驗證 H_BY_VAR（或等價 adjacency-by-variable）解讀與設計端 check_parity 更新結果一致。
- 至少要有小模型 / 小矩陣 case 驗證 parity accumulator 的正確性。
- 不允許 TB 用第二套與 spec 無關的 H 解碼規則繞過正式 layout semantics。

15.6 base mask / runtime derived rule 驗證
- TB 必須驗證 base mask 本體、runtime ring1 / ring2 派生，以及 head-group 切換規則。
- 不允許 TB 直接餵兩張已展開的 ring masks 來繞過設計實作。
- 若 softmax 參考模型仍沿用 two-pass，必須在 report 明列為與 v12.1 spec 不對齊項目，不得默認視為相容。

====================================================================
16. Legacy compatibility 備註
====================================================================


16.1 舊版 split bias 流程
- LOAD_BIAS / BIAS_RX / WEIGHT_RX 可為 backward compatibility 保留，但 unified PARAM 才是預設。
- 若保留此路徑，BiasDesc.h（或等價 bias metadata）與 WeightStreamOrder.h 必須繼續作為 bias/weight expected_len 的唯一來源。

16.2 舊版 dual-port 實體敘述
- 歷史文件中提到的單顆 2-port SRAM 或 bank-lock 風格，只保留為背景資訊。
- 目前唯一有效的實作模型，是本 spec 所定義的 single-port deterministic 1-issue 架構。

16.3 歷史 append-only 咒語
- 歷史咒語檔依然具有 archive / audit trail 價值。
- AECCT_HLS_Spec 是後續實作與維護應使用的乾淨主規格。
