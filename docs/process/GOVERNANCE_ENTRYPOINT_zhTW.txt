GOVERNANCE_ENTRYPOINT_zhTW.txt
================================

文件定位
--------
本文件是 AECCT_HLS 專案的「治理入口文件」。
用途是讓新對話、新任務、新接手者能在最短時間內知道：

1. 先看哪幾份文件
2. 每份文件各自負責什麼
3. 開新 Codex / ChatGPT 任務時，最少要附哪些文件
4. 驗收時最低要看哪些證據
5. 哪些文件屬於日常核心，哪些屬於治理查核時再讀

目前工作重點
--------
目前 reviewer-facing debt/paydown 已完成核心收斂，已由 blocking 轉入維護期；目前重點改為：
- 維持文件結構清楚（process/review/handoff/archive 分層）
- 保持 reviewer kit 可快速導航、可重用
- 讓 execution PASS 與 reviewer-understood 兩件事持續分開管理（避免 overclaim）
- 後續若繼續補文件，定位為 optional enhancement（例如額外情境模板、增量 quick-check sidecar），不再視為 core outstanding debt
- functional/mainline work 可與文件維護並行推進，不必再把 readability backfill 當成唯一前線優先順序

本文件不是：
- 專案目前狀態摘要（那是 PROJECT_STATUS_zhTW.txt）
- 任務下法與驗收細則全文（那是 AECCT_PROJECT_WORKFLOW_v1_zhTW.txt）
- HLS / Catapult 實作規範（那是 Catapult_C++_CodeGen_Guide_for_Codex_v3.txt）
- 長期里程碑規劃（那是 AECCT_v12_M0-M24_plan_zhTW.txt）

若你要開始一個新任務，建議先依下列順序對齊：

4. docs/spec/AECCT_HLS_Spec_zhTW.txt
   - 用途：主規格、正式 contract、正式硬體行為定義

5. docs/architecture/AECCT_HLS_Architecture_Guide_zhTW.txt
   - 用途：邏輯資料流、實體落點、micro-architecture 收斂參考

6. docs/architecture/AECCT_module_interface_rules_zhTW.txt

4. 任務啟動最低要求（摘要版；完整模板請看 WORKFLOW）
====================================================================

每次交給 Codex / ChatGPT 的任務，至少應明確寫出：

1. Goal
   - 這次要完成什麼

2. Scope In
   - 這次允許做的事

3. Scope Out
   - 這次不允許做的事

4. Authority Bundle
   - 這次要依循哪些文件 / artifact / metadata

5. Definition of Done（DoD）
   - 什麼情況才算完成

6. Deliverables
   - 最後必須回報哪些東西

7. Repro / Test Plan
   - 怎麼 build / run / 驗證

詳細寫法請回到：
- AECCT_PROJECT_WORKFLOW_v1_zhTW.txt

====================================================================
5. 驗收最低要求（摘要版；完整驗收分級請看 WORKFLOW）
====================================================================

本專案對 code 任務的最低驗收規則如下：

完成 ≠ 文字摘要完成

凡是 code change / patch / checker / preview / milestone 子任務聲稱完成時，
至少必須同時看到：

1. Files changed
2. Build / run command
3. Actual execution evidence

若沒有 execution evidence，
最多只能算：
- 聲明
- 初步回報
- 摘要合理

不能直接算正式完成。

補充：
- 小任務 / checker / preview 可採輕量治理
- 但只要是 code 任務，execution evidence 仍屬驗收必要項
- 若尚未執行，必須明確寫：
  - not executed

詳細規則請回到：
- AECCT_PROJECT_WORKFLOW_v1_zhTW.txt

補充：execution evidence 只回答「有沒有跑」，不自動回答「reviewer 是否已理解」。
若任務涉及 design-side code，後續仍應搭配 reviewer guide / readability sidecar / loop-role 說明。

7. ChatGPT / Reviewer 的固定判讀口徑
====================================================================

7.1 不能混淆「驗證基礎設施進展」與「主路徑功能進展」
----------------------------------------------------
例如：
- 新增 checker / preview / parser validation
屬於：
- 驗證基礎設施的進展

不代表：
- live migration 已完成
- main path 已接好
- numeric closure 已完成

7.2 證據不足時的正確說法
------------------------
若只有摘要、沒有 build/run evidence，
應明確說：
- 目前只能判斷方向合理
- 目前只能判斷摘要與 spec 大致一致
- 不能直接判定為正式完成

7.3 不要把 scope 守住，誤解成主系統功能前進很多
-----------------------------------------------
例如：
- 只新增 1 支 checker TB
可以代表：
- 該子任務範圍控制正確
- preview / checker 基礎設施前進了

但不能自動代表：
- live path 更完整
- main system 已大幅前進

10. 一句話版核心原則
====================================================================

本專案的治理入口原則是：

先用 PROJECT_STATUS 對齊當前位置，
再用 PROJECT_WORKFLOW 對齊任務與驗收規則，
若涉及 code 再用 CodeGen Guide 對齊寫法，
若涉及正式規格與記憶體語意，再補 spec / architecture / rules，
而所有 code 任務的完成都必須有 execution evidence，不能只看摘要。
