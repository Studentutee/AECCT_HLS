Codex 咒語注意事項（總規範）
File: Codex_Spell_Guide_v1.1.txt
Version: Codex_Spell_Guide_v1.1 (2026-03-06)  Encoding: UTF-8 (NO BOM)

本文件目的：把「BUILD / PATCH / REFACTOR 咒語」整理成一套可控、可驗證、可回滾的操作規範，
讓你用 Codex 生成/修改 AECCT_HLS 時：
- 不會亂改前面里程碑的資料（Scope Lock + 可機械化 Gate）
- 不會走錯目前 v12 的硬體架構（Spec 鎖定）
- 不會嘴上說 PASS（Evidence Pack + 可重現 Repro）
- 能快速回溯與回滾（小 diff、可追溯 artifacts）

本文件的定位：本文件只規範「如何下咒語、如何要求 Codex 工作」。
它不是硬體規格本體，也不是治理文件本體；它負責把那些文件正確地串起來。

====================================================================
1) 三種 MODE（每次咒語必須明確選一種）
====================================================================

MODE=BUILD
- 用途：新增檔案/新里程碑（M0/M1/M2…），或把缺的 skeleton 補齊。
- 特性：允許新增較多檔案，但仍要 Scope Lock；禁止偷偷改 Contract/SSOT；禁止順便大重構舊檔。
- 典型情境：新增 Top skeleton、新增 block、新增對應 TB、補齊某 milestone 缺的 headers。

MODE=PATCH
- 用途：修 bug、合規收斂、改一個明確問題。
- 特性：最小 diff、一次只解一個問題（One patch, one problem）。
- 典型情境：修 bitfield、修 state transition、修 READ_MEM、修某個 HLS-safe 違規點。

MODE=REFACTOR
- 用途：允許搬檔/重命名/結構重整（例如把 include/gen/design/tb 切得更乾淨）。
- 特性：diff 會較大，但需要更嚴格的 migration plan、rollback plan、Evidence Pack、Gate。
- 典型情境：調整 repo 結構、把 legacy headers 收斂成 SSOT、拆分 spec/governance 類文件。

====================================================================
2) 每次咒語都必須自包含（Self-contained）
====================================================================

你不能假設 Codex 記得上下文；每次咒語都要至少帶這些欄位：

(2.1) Identity
- MODE
- Patch/Milestone/Refactor ID
- 目標里程碑（若適用）
- Baseline（branch/commit/zip）

(2.2) Authority Bundle（文件版本束）
至少要明確指向：
- Spec version
- Governance version
- CodeGen Guide version
- MasterPlan version（若本次和里程碑直接相關）

(2.3) Goal
- 一句話描述目標（What to achieve）

(2.4) Why
- 為什麼要做：功能、合成性、規格一致性、驗證、性能、repo 治理

(2.5) Definition of Done（DoD）
- 哪些 TB 必須 PASS
- 哪些 Gates 必須 PASS
- 哪些交付物必須存在
- 是否允許 Known limitations（若允許要明說）

(2.6) Repro
- Build 指令（含 working dir / include path / defines）
- Run 指令
- 目前 FAIL signature（若是 patch）

(2.7) Scope Lock
- ALLOW：這次可以改哪些檔案/目錄
- DENY：這次不能改哪些檔案/目錄
- 若需擴張白名單，必須在 report 記錄理由

(2.8) Locks
- Contract Lock
- SSOT Lock
- Boundary Lock
- Architecture Lock（例如 current memory model = single-port deterministic 1-issue）

(2.9) Deliverables
- report
- artifacts
- diff.patch
- build.log / run_tb.log
- verdict.txt
- file_manifest.txt

====================================================================
3) 共通硬規則（Common Hard Rules / MUST）
====================================================================

(3.1) 正式來源不要寫錯
- 任何外部可觀察契約（4-channel、bitfield、rsp 順序）一律以 Spec 為準
- 任何 README / Gate / boundary / hygiene / packaging 規則一律以 Governance 為準
- 任何 Catapult/CCS 寫法一律以 CodeGen Guide 為準
- 任何 milestone 定義與 DoD 一律以 MasterPlan 為準

(3.2) Encoding & Comments
- 所有產出檔案：UTF-8（NO BOM）
- 程式碼註解（.h/.cpp/.cc/.hpp）：English only
- README / docs 可中文，但避免亂碼

(3.3) Contract / SSOT 不得偷偷改
Contract 至少包含：
- Top 4 channels 型別與命名
- ctrl_cmd / ctrl_rsp bitfield
- ctrl_rsp 正常回應順序（成功路徑 OK → DONE；錯誤路徑 ERR(err_code)）
SSOT 至少包含：
- opcode / errcode / rsp pack-unpack 集中在 include/ 的單一位置
- shape / expected_len / SramMap / WeightStreamOrder 各有唯一來源
- Top/TB/blocks 不可各複製一份

(3.4) 架構前提不要用舊版假設
- 目前 v12 的主架構前提，以 Spec 為準
- 對 memory / arbitration / overlap / FinalHead / debug / quant 的描述，不得再引用已失效的舊前提
- 特別注意：不要再對 Codex 下達 dual-port / 2-issue 的假設，除非你明確要開新版本規格

(3.5) DESIGN vs TB boundary
- 正式規則與 Gate 來源：Governance
- 在咒語中至少要重申：
  - DESIGN 禁 trace/golden/step0 headers
  - TB 才能 include trace 與 golden
  - 不准把 TB-only helper 滲進 DESIGN

(3.6) Weights / Params boundary
- 正式規則與 Gate 來源：Governance
- 在咒語中至少要重申：
  - DESIGN 不得依賴 compile-time weights 作為運行期權重來源
  - 正式權重來源必須是參數載入後的 SRAM / PARAM region
  - 若本次 patch 涉及 weights path，必須檢查 Design/TB boundary 與 check_design_purity

(3.7) HLS-safe subset
- 正式來源：CodeGen Guide
- 在咒語中至少要重申：
  - DESIGN 禁 C++ float/double（除非規格與 Guide 明確允許特例）
  - DESIGN 禁 union/memcpy bit-cast、<cmath>、new/delete、exceptions、RTTI、iostream/fstream、host file IO、重度 STL
  - DESIGN 優先 ac_int / ac_fixed / ac_ieee_float<binary32>

(3.8) 沒有證據，不准宣稱 PASS
- 沒有 build/run logs、沒有 verdict、沒有 diff.patch、沒有 report，一律不得宣稱 PASS
- Gate FAIL 時，不得以「TB 看起來過了」宣稱里程碑已完成

====================================================================
4) Evidence Pack 與 Gate 的寫法（由 Spell Guide 轉接到 Governance）
====================================================================

(4.1) Evidence Pack 的正式定義
- 以 Governance 的「里程碑交付證據包 / 補丁交付證據包」為準
- 本文件只要求：每次咒語都要明確要求 Codex 產出那些檔案，並把路徑寫清楚

(4.2) Gates 的正式定義
- 以 Governance 為準
- 本文件建議在每次咒語至少列出：
  - check_design_purity
  - check_interface_lock
  - check_repo_hygiene
  - check_scope_lock（若有）
  - check_tb_flow（若本次屬於 bring-up / top flow）
- 若你這次是 REFACTOR，可再加 refactor-specific gates

(4.3) run_gates wrapper
- 建議要求 Codex 統一使用 scripts/run_gates.sh 或 scripts/run_gates.ps1
- wrapper 必須用非 0 exit code 阻止「嘴上 PASS」

====================================================================
5) Scope Lock（修改範圍鎖）— 每次咒語都要列
====================================================================

(5.1) 白名單（ALLOW）要越小越好
- 從 1~3 個檔案起步最穩
- 若不得不擴大，必須在 report 的 What changed 記錄原因

(5.2) 黑名單（DENY）建議每次都列
至少建議列出：
- Top 介面/bitfield 的 Contract 檔
- opcode / errcode / rsp pack-unpack 的 SSOT 檔
- ModelShapes / ModelDesc / SramMap / WeightStreamOrder 等 SSOT 檔（若本次不是要改它們）
- trace/golden/data dumps
- governance/spec/codegen/plan 中本次無關的文件

(5.3) 當你其實是在改規格，不要假裝是 patch
- 如果需要改 Contract、memory model、FinalHead 行為、quant 基線、error code 定義，這通常不是 PATCH
- 你應該先改 Spec / Governance / Plan，再要求程式配合

====================================================================
6) BUILD Mode 細則
====================================================================

BUILD Mode 的重點是「把新東西蓋出來，但不要把舊東西踩壞」。

(6.1) BUILD 咒語必備
- 目標 milestone / 子目標
- 要新增哪些檔案（檔名、目錄、用途）
- 每個新檔的 API / class / function 骨架
- 依賴哪份 Spec / 哪個 milestone / 哪些 SSOT headers
- 需要哪些 TB / smoke test
- Scope Lock
- Deliverables / Gates

(6.2) BUILD 不得做的事
- 不准順便大重構舊檔
- 不准默默改 Contract / SSOT
- 不准把 legacy 行為偷偷清掉而不更新 Spec / Governance / Plan

(6.3) BUILD 建議
- 直接引用 MasterPlan 中的 milestone 條目
- 對 Top / blocks / TB 的程式架構，直接引用 CodeGen Guide
- 對 memory / protocol / debug / quant 的行為，直接引用 Spec

====================================================================
7) PATCH Mode 細則
====================================================================

PATCH 的精神仍然是：One patch, one problem。

(7.1) PATCH 三段式流程
Phase 1 — Preflight
1) 列出本次 Scope
2) 列出不會被改的 Contract / SSOT / boundary
3) 先跑 baseline build / run / gates，保留 FAIL 證據
4) 在咒語中釘死 Authority Bundle（Spec / Governance / CodeGen / Plan）

Phase 2 — Minimal Change
1) 一次只修一件事
2) 禁止順便 rename / 大格式化 / 搬檔 / 風格重排
3) 若真的需要擴 Scope，先在 report 說明

Phase 3 — Verify & Evidence
1) 跑 TB
2) 跑 Gates
3) 產生 diff.patch
4) 產生 verdict.txt
5) 產生 report + artifacts

(7.2) PATCH 咒語適合處理的問題
- 某個 state transition 錯
- 某個 bitfield / rsp packing 錯
- 某個 READ_MEM / DEBUG_CFG 行為不符 Spec
- 某個 HLS-safe 違規點（例如誤用 host math）
- 某個 boundary/gate/repo hygiene 問題

(7.3) PATCH 不適合處理的問題
- 要整體改 memory model
- 要換整套 quant / softmax / FinalHead 規格
- 要改 milestone 定義
- 要把 repo 結構大搬家

====================================================================
8) REFACTOR Mode 細則
====================================================================

REFACTOR 不是禁止，而是要更嚴格。

(8.1) REFACTOR 咒語必備
- Migration plan：要搬什麼、為什麼
- Compatibility plan：include path / build scripts / docs 怎麼維持可用
- Rollback plan：如何快速退回舊狀態
- Refactor-specific gates
- 最少一個 smoke TB + 全部必要 gates

(8.2) 常見適合 REFACTOR 的情境
- include/gen/design/tb 重新分層
- SSOT headers 收斂
- spec/governance/plan 文檔重整
- 把 legacy 路徑與正式路徑切開

(8.3) REFACTOR 時一定要小心
- 不要藉 refactor 偷改硬體行為
- 若硬體行為被改到，先更新 Spec
- 若治理規則被改到，先更新 Governance
