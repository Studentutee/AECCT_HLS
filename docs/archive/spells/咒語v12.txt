咒語 v12（在 v11.12 基礎上增補；不刪舊內容）
Version: 2026-03-06

目的：維持 v11.12 與下方所有 legacy 段落「原樣保留」，並新增/覆寫以下條款（本段優先於後續任何舊段落）。

====================================================================
【v12 增補條款：預留資料量化規劃（pre-infer configurable；grouping/zone 尚未定版）】
====================================================================

(0) 本版定位（先把計畫寫進規格，但暫不強迫實作落地）
- 本版正式記錄：後續版本預計導入「資料量化」規劃。
- 本版的目標是把量化方向、邊界條件、SSOT/命令預留、驗證規則先寫進咒語，讓後續版本可無痛擴充。
- 本版【不要求】目前所有 datapath 立即改成量化實作。
- 本版【不要求】目前就凍結最終 grouping / zone 分區方式。
- 本版【不要求】目前就決定所有 block 的最終 exponent / scale 配置。
- 本版【不改】既有 4 channels contract、既有 CFG/PARAM/INFER 基本流程、既有 FinalHead / X_PAGE / SCRATCH / overlap / P-1/P-2 / single-port deterministic 1-issue 等規格。

(1) 規劃方向（先記錄設計意圖）
- 本版正式記錄之量化規劃方向為：
  - 以「整數值（mantissa/int payload） + 指數（exponent）」作為主要探索方向。
  - exponent/scale 的設定屬於「推論前配置（pre-infer configurable）」：在進入一次 INFER 前可先配置；同一次 inference 期間不允許任意改變。
  - 目前要解的是：先用離線數值分析（distribution / MSE / error study）找出較佳配置，再決定 grouping 與配置表；不是做 sample-by-sample 動態浮動格式。
- 本版不把此規劃過度綁死為特定數學表示；若後續實證顯示需在 implementation 上以 fixed-point / int+exp hybrid / boundary-requant 方式落地，允許後續版本補充，但不得違反本版之「pre-infer configurable、非 inference 途中任意改變」原則。

(2) 重要邊界（先釘死，避免日後語意漂移）
- 「動態」在本規劃中的意思，一律指：
  - 在一次 inference 開始前可由 host/config 寫入一組量化配置。
  - 一旦該次 inference 開始，該組配置在本次 inference 期間視為固定。
- 本規劃【不是】：
  - 每個 sample / token / 向量 / 中間值在執行途中各自切換不同格式。
  - 每筆資料都各自攜帶獨立 exponent 的一般浮點流程。
  - 為了量化而破壞既有 deterministic scheduling / memory contract。
- 若未來導入 shared exponent / grouped exponent，其語意也屬於「pre-infer loaded configuration」，不是 runtime free-running adaptation。

(3) grouping / zone 分區：本版正式保留，暫不定版
- 本版承認：量化 grouping / zone 分區方式目前【尚未定版】。
- 也就是說，現階段不在咒語中強制寫死：
  - zone 的最終數量
  - zone 的最終命名
  - zone 是否依 layer type、模塊邊界、consumer-side domain、或其他規則分組
  - zone payload 的最終欄位數
- 但本版要求：後續若要定版 grouping，必須以「不破壞既有外部 contract」為優先，並盡量透過新增 SSOT / 新增配置表 / 新增保留命令達成，而非回頭大改既有推論流程。
- 本版允許後續版本從以下方向中擇一或混合定版（僅作方向記錄，不代表本版已定案）：
  - 依模塊邊界輸入/輸出（module boundary I/O domains）
  - 依 consumer-side numeric contract
  - 依 layer / sublayer 類型
  - 依 persistent buffer / major datapath lifetime
- 在 grouping 未定版前，任何新檔案/新程式/新 TB 都不得擅自把 zone count 或 zone enum 寫成硬規格。

(4) SSOT 預留：QuantDesc / QuantConfig 類檔案
- 本版要求：後續量化規劃若正式導入，必須有獨立 SSOT 檔案（例如 `QuantDesc.h`、`QuantDesc_gen.h`、`QuantConfig.h` 或等價命名）。
- 該 SSOT 的責任預期包括（本版僅預留，不要求立刻完整實作）：
  - 記錄量化模式版本（version）
  - 記錄 grouping / zone 定義（待定版後）
  - 記錄每一組的 exponent / scale / 或等價配置
  - 記錄合法範圍、預設值、與 host payload decode 規則
- 在本版階段，若專案中尚未實作該 SSOT 檔案，允許先以註解 / placeholder / TODO 形式預留，但不得把量化規劃只散落在 TB 或臨時 script 中。

(5) 配置命令 / payload 預留（本版先保留，不強制釘死 opcode）
- 本版正式記錄：後續量化配置應有獨立的配置入口（例如保留命令 `SET_QCFG` 或等價機制）。
- 但本版【不強制】立即定案：
  - opcode 數值
  - payload 長度
  - payload 欄位定義
  - 是走 data_in 串流、CFG 擴充、或其他相容方式
- 本版僅要求：
  - 後續若導入量化配置入口，必須維持與既有 4-channel contract 相容。
  - 後續若導入量化配置入口，其配置時機必須在無 in-flight inference 時進行；不得在同一次 inference 中途任意重寫。
  - 若未來定義了 `SET_QCFG`，其 busy / state 規則應與既有 CFG/PARAM 類命令保持一致精神（避免 overlap 下配置污染）。

(6) 實作邊界（避免現在就把設計鎖死）
- 本版不要求目前實作：
  - 每個 block 內部的 local temporary / accumulator 都受全域量化配置支配。
  - 所有 debug trace node（例如 *_act_q / *_out_q）直接成為 zone 定義依據。
  - 所有 scale / reciprocal / normalization 中間值立即納入同一套全域配置表。
- 本版允許後續版本採取：
  - 全域可配置 + block local fixed handling 的混合式設計
  - 先只配置 major persistent domains，再逐步擴充
  - 先做分析與 TB 驗證，再逐步把 datapath 接上
- 本版的原則是：先把 architecture hook / spec hook 留好，再決定最終 grouping 與硬體化細節。

(7) 離線分析流程：正式列入規劃
- 本版正式承認：量化配置的來源可來自離線分析流程（例如 distribution / MSE / compare traces / weight sweep / exponent sweep / quant search）。
- 後續版本允許/鼓勵：
  - 由 script 讀取 trace / weights / reference tensors
  - 產生建議 exponent / scale / grouping 報表
  - 再由生成器或轉檔流程產出 QuantDesc / 預設 payload
- 但在本版階段：
  - 離線分析結果僅作為「規劃與決策依據」
  - 不得在咒語中假裝 grouping 已定版
  - 不得把目前暫時觀察到的群組直接寫死為永遠的硬規格

(8) 里程碑與凍結時機（先記錄，不強迫當下完成）
- 本版建議把量化導入拆成至少兩個階段：
  - 第一階段：規格預留 / SSOT 預留 / 命令預留 / 分析流程建立
  - 第二階段：grouping 定版 / payload 定版 / TB 對應 / datapath 真正接線
- 在 grouping 未定版前：
  - 咒語可以明寫「有量化計畫」
  - 但不得要求 Codex / 產碼端現在就依未知 zone enum 生成最終版硬體
- 建議在後續某一版再做 quant spec freeze；freeze 後才允許把 group id / payload layout / decode table 寫死。

(9) TB / 驗證預留
- 本版要求：後續若導入量化配置，TB 必須能區分：
  - 未啟用量化配置（legacy path）
  - 啟用量化配置但 grouping/設定為測試版
  - grouping / payload 定版後的正式量化路徑
- 在 grouping 未定版前，本版不要求立即新增完整量化 TB matrix。
- 但允許/鼓勵在文件中預留：
  - quant-enabled regression
  - 比對 reference / tolerant compare / MSE report
  - payload legality / busy-state / freeze-time checks

(10) 優先級與相容
- 本 v12 條款的功能是「記錄與預留量化規劃」，不是推翻 v11.12 既有 FinalHead / page / scratch / streaming 規格。
- 若後續版本導入量化實作，仍必須與 v11.12 以及下方所有 legacy 條款相容；如有衝突，需以新增條款明確覆蓋，不得默默改語意。
- 本版新增條款若與下方舊段落衝突，以本 v12 增補條款為準；未衝突者全部沿用既有內容。

咒語 v11.12（在 v11.11 基礎上增補；不刪舊內容）
Version: 2026-03-06

目的：維持 v11.11 與下方所有 legacy 段落「原樣保留」，並新增/覆寫以下條款（本段優先於後續任何舊段落）。

====================================================================
【v11.12 增補條款：FinalHead 的 s_t 固定落在 SCRATCH；logits 直接 streaming 輸出；FinalHead 不得使用 PAGE_NEXT】
====================================================================

(0) 名詞與形狀（避免模型結構過定）
- 所有 shape（T_TOKENS、D_MODEL、N_LAYERS、OUT_DIM、N_VARS…）皆以 CFG / ModelShapes / ModelDesc 的 SSOT 為準，禁止在規格或程式中寫死 75×32、63、或任何固定 token 數。
- X 向量 shape 一律以 x[token][d] 表示：token 是列(Row)，d 是行(Colume)；線性化規則仍為 d/行(Colume) inner-most contiguous。
- FinalHead 的 token-wise scalar 記為 s_t，其中 t=0..T_TOKENS-1；s_t 的長度隨模型而變，因此不得假設為固定大小 regs-only buffer。

(1) 本版設計決策（覆蓋 v11.11 的 TEMP_PAGE readout 配置）
- 保留 v11.11 的 final-layer no-writeback：
  - 對最後一層 layer = N_LAYERS-1，仍然禁止把 x_last[token][d] 寫回 next_page。
- 但覆蓋 v11.11 對 next_page / TEMP_PAGE 的 readout 用法：
  - FinalHead 的 s_t 暫存不得放在 next_page / TEMP_PAGE。
  - FinalHead 的 logits 不得配置 OUT_LOGITS_BUF 落在 next_page / TEMP_PAGE。
  - FinalHead 在本版中完全不使用 PAGE_NEXT 作為 readout 暫存。
- 理由：PAGE_NEXT 必須保留給下一次 input / staging / 後續 page 生命週期管理，避免與 FinalHead 的短生命週期暫存發生 liveness 衝突。

(2) 新規則：s_t 必須落在 SCRATCH SRAM（可變長度；非 regs-only）
- 新增 / 覆寫 readout 規則：
  - s_t 必須寫入 SCRATCH 內的 FINAL_SCALAR_BUF。
  - FINAL_SCALAR_BUF 的容量必須依 runtime cfg 決定，因為 s_t 長度與 T_TOKENS（或等價 cfg_n_nodes）相關。
- FINAL_SCALAR_BUF 定義：
  - 語意：存放每個 token 對應的一個 scalar s_t
  - shape：FINAL_SCALAR_BUF[t]，t=0..T_TOKENS-1
  - words：ST_WORDS = align_up(T_TOKENS, ALIGN_WORDS)
- FINAL_SCALAR_BUF 屬於 SCRATCH lifetime，不屬於 X ping-pong page，不得被宣告為 X_PAGE 的子區或 page tail。

(3) FinalHead 的輸出策略：logits 直接走 streaming，不落 SRAM
- FinalHead 的最終 logits / x_pred 仍以 output channel 為正常輸出路徑。
- 本版新增硬規則：
  - logits 不配置 OUT_LOGITS_BUF 作為正常輸出 staging。
  - Pass B 產生的 logits[c] 必須直接依 CFG_OUT_MODE / OUTMODE 送到 data_out。
  - 不得要求 host 以 READ_MEM 讀回 OUT_LOGITS_BUF 才取得正常推論輸出。
- 若 CFG_OUT_MODE = logits：依序 streaming 輸出 OUT_DIM 筆 logits。
- 若 CFG_OUT_MODE = x_pred：以既有 logits→x_pred 的規則在 FinalHead 內產生後，依序 streaming 輸出。

(4) FinalHead Two-pass Readout（本版正式定義）
Pass A（token-wise）
- 讀最後一層可見的 x_last[token][0:D_MODEL]（來源仍依 v11.11 的 final-layer no-writeback 規則）
- 計算：s_t = FinalEmbedding(LN(x_last[token][0:D_MODEL]))
- 將 s_t 寫入 SCRATCH.FINAL_SCALAR_BUF[t]

Pass B（GEMV / reduction）
- 讀 SCRATCH.FINAL_SCALAR_BUF[t]
- 計算：logits[c] = Σ_t OUT_FC[c,t] * FINAL_SCALAR_BUF[t]，c=0..OUT_DIM-1
- logits 不落地到 SRAM；每一筆（或每個 tile 完成後）直接往 data_out streaming 輸出
- 若需要 class tiling（例如 TILE_C=8/16），只允許 tile accumulator 以 regs / local buffer 暫存；tile 完成後直接輸出，不得因此新增 OUT_LOGITS_BUF 的正常流程依賴。

(5) PAGE_NEXT / TEMP_PAGE 的禁止事項（本版硬規則）
- FinalHead 階段：
  - 不得把 s_t 寫入 PAGE_NEXT
  - 不得把 FINAL_SCALAR_BUF 配到 PAGE_NEXT
  - 不得把 logits / OUT_LOGITS_BUF 配到 PAGE_NEXT
  - 不得宣稱 next_page 為 FinalHead 專用 TEMP_PAGE
- 也就是說：v11.11 中「next_page 在最後一層期間可被切成 FINAL_SCALAR_BUF / OUT_LOGITS_BUF」的條文，對本版 FinalHead 正常路徑不再適用。

(6) SCRATCH 的生命週期（liveness）與 overlap 規則
- FINAL_SCALAR_BUF 的 liveness 僅限於同一個 inference 的 FinalHead Pass A / Pass B 期間。
- FinalHead 完成後，FINAL_SCALAR_BUF 可被覆寫，不得被視為 persistent output。
- 若啟用 multi-context overlap：
  - 只有在兩個 context 可能同時活到 FinalHead 且需要並存 FINAL_SCALAR_BUF 時，才要求 per-context duplication。
  - 若實作上保證同一時間只會有一個 context 進入 FinalHead，則可維持單份 FINAL_SCALAR_BUF，但行為必須 deterministic，且不得造成另一個 context 的 scratch 資料被誤覆寫。

(7) SramMap.h / 生成器 / 配置的新增要求
- SramMap.h（或等價的離線生成輸出）必須新增 / 提供：
  - SCR_FINAL_SCALAR_BASE
  - SCR_FINAL_SCALAR_WORDS
- 若支援 overlap 且 FinalHead scratch 需並存，可選擇擴充為：
  - SCR_FINAL_SCALAR_CTX0_BASE / WORDS
  - SCR_FINAL_SCALAR_CTX1_BASE / WORDS
- 生成器必須以 runtime cfg / ModelShapes 規則決定 ST_WORDS，不得寫死固定長度。

(8) TB / 驗證規則（必須同步更新）
- OUTMODE_LOGITS：TB 必須以 data_out 讀取 streaming logits 做比對；不得把 READ_MEM(OUT_LOGITS_BUF) 當正常驗收路徑。
- OUTMODE_XPRED：TB 必須以 data_out 讀取 streaming x_pred 做比對。
- 若要 debug / inspect s_t：
  - 允許用 READ_MEM 讀 SCRATCH.FINAL_SCALAR_BUF
  - 但此路徑僅屬 debug / verification，不屬正常輸出 contract。
- 若啟用動態 debug/halt（D1），仍遵守既有 HALTED + READ_MEM 規則；本版只改 FinalHead 正常 readout 落點，不改 debug contract。

(9) Gate / 驗收（避免退化）
- 必須能證明 FinalHead 正常路徑不使用 PAGE_NEXT 作為 s_t 或 logits buffer。
- 必須能證明 FINAL_SCALAR_BUF 的落點位於 SCRATCH region。
- 必須能證明正常 logits/x_pred 輸出由 data_out 自動輸出，而非要求 host 追加 READ_MEM。
- 若程式 / 文件仍保留 v11.11 關於 TEMP_PAGE readout 的舊描述，於 FinalHead 正常路徑有衝突者，以本 v11.12 增補條款為準。

咒語 v11.11（在 v11.10 基礎上增補；不刪舊內容）
Version: 2026-03-06

目的：維持 v11.10 與下方所有 legacy 段落「原樣保留」，並新增/覆寫以下條款（本段優先於後續任何舊段落）。

====================================================================
【v11.11 增補條款：Page base swap + Final-layer no-writeback + Two-pass Readout（OUT_DIM 可變）】
====================================================================

(0) 名詞與形狀（避免模型結構過定）
- 所有 shape（T_TOKENS、D_MODEL、N_LAYERS、OUT_DIM…）皆以 CFG/ModelShapes/ModelDesc 的 SSOT 為準，禁止在規格或程式中寫死 75×32 或 63。
- X 向量 shape 一律以 x[token][d] 表示：token 是列(Row)，d 是行(Colume)；線性化規則：d/行(Colume) 為 inner-most contiguous。

(1) X_PAGE 的本質：同一塊 SRAM 內的「base 指標」切換（swap=切 base，不搬資料）
- X_PAGE0 / X_PAGE1 不是固定用途的兩頁，而是同一個 X_REGION 內的兩個 base：
  - PAGE_WORDS = align_up(T_TOKENS * D_MODEL, ALIGN_WORDS)
  - PAGE0_BASE = X_REGION_BASE + 0 * PAGE_WORDS
  - PAGE1_BASE = X_REGION_BASE + 1 * PAGE_WORDS
- runtime 只維護 (active_base, next_base)，swap_pages() 僅交換 base（或切換 index），禁止任何 page-to-page memcpy。

(2) Layer 排程：前面層照常寫回 next_page；最後一層禁止寫回 X_PAGE（final-layer no-writeback）
- Preproc 會把 token embedding（input_y*embed 與 lpe_token 拼接後的 x[token][d]）寫入 active_page。
- 對 layer = 0 .. N_LAYERS-2：
  - read active_page → compute → write next_page → swap_pages()
- 對最後一層 layer = N_LAYERS-1：
  - read active_page → compute（token-wise streaming）
  - 【禁止】把最後一層輸出 x_last[token][d] 寫回 next_page
  - next_page 在最後一層期間視為 TEMP_PAGE（可被切割成暫存區或 CTX_NEXT staging；見 (4)）

(3) READ_MEM 的地位：debug/驗證用；正常輸出不依賴 READ_MEM
- READ_MEM 仍可用於 debug（例如 checkpoint/對齊），但不得把 READ_MEM 當作正常輸出機制。
- 正常輸出行為由 CFG_OUT_MODE（或等價的 OUTPUT_SEL/OUTMODE）決定：推論完成（INFER_DONE）後自動輸出，不需要任何額外「輸出指令」。

(4) OUT_DIM 可變 + logits 需要累加：採用 Two-pass Readout（內部 SRAM 暫存），最終仍自動輸出
- 動機：OUT_DIM 可變時，logits accumulator 不可過定成固定大小 reg array；因此允許/鼓勵用 SRAM 做可調長度的累加/存放。

Two-pass Readout 定義：
Pass A（token-wise，與最後一層 token 流水線同時進行）
- 對每個 token t（列(Row)）：
  - 計算 s_t = FinalEmbedding(LN(x_last[t][0:D_MODEL]))，s_t 為 scalar
  - 將 s_t 寫入 FINAL_SCALAR_BUF[t]

Pass B（GEMV，產生 logits）
- logits[c] = Σ_t OUT_FC[c,t] * FINAL_SCALAR_BUF[t]，c=0..OUT_DIM-1
- 使用 class tiling（TILE_C=8/16 建議）：
  - acc_tile[TILE_C] 用 regs 累加
  - tile 完成後一次性寫回 OUT_LOGITS_BUF[c_tile]
- 完成後依 CFG_OUT_MODE 自動輸出：
  - logits 模式：依序輸出 OUT_DIM 筆 logits
  - x_pred 模式：以 logits 與 input_y 的 sign/既有規則產生 bit，依序輸出（輸出筆數視模型定義，通常為 N_VARS）

(5) TEMP_PAGE 的切割（避免過定，保留彈性）
- final layer 不寫回 next_page，因此 next_page 可被切成多段暫存：
  - FINAL_SCALAR_BUF：至少容納 T_TOKENS 個 scalar
  - OUT_LOGITS_BUF：至少容納 OUT_DIM 個 scalar（若 Pass B 需要落地）
  - 剩餘空間（若有）允許用於 CTX_NEXT 的 preproc staging（選配 overlap），或其他 scratch，但需遵守 liveness（不得覆寫仍需資料）。
- 必須有容量檢查：
  - FINAL_WORDS + OUT_WORDS <= PAGE_WORDS 時，允許兩者都放在 TEMP_PAGE
  - 若 OUT_DIM 過大導致塞不下，OUT_LOGITS_BUF 必須改放到既有 OUT_REGION/SCRATCH 中由 SramMap/CFG 給定位址

(6) Gate/驗收（釘死避免退化）
- 必須證明 final layer 沒有對 next_page 做任何 full-page writeback（可用 code audit 或簡單計數器）。
- CFG_OUT_MODE 決定自動輸出；不得要求 host 透過 READ_MEM 才拿到最終結果。
- OUT_DIM 可變：Pass B 必須以 OUT_DIM 為迴圈界線，禁止硬寫 63。

【W) v11.10 新增：Per-layer Dynamic Quant ScaleTable（s_x + inv_s_x）+ Scale Estimator（必做）】
====================================================================

[W-0] 覆蓋範圍（本條款優先）
- 本條款新增「動態量化尺度」行為，不改動既有 4 channels contract、不新增 opcode、不新增任何外部欄位。
- 本條款與 v11.9 single-port/1-issue 仲裁相容：所有新增的 SRAM 存取仍受 [S-*] deterministic priority 約束。

[W-1] 名詞與基本定義
- s_x：量化尺度（scale）。
- inv_s_x：s_x 的倒數（English: reciprocal / gain），inv_s_x = 1 / s_x。
- 量化建議主路徑（示意）：q = round(x * inv_s_x) + clamp/sat。
- slot（每層固定 4 個；與既有命名對應）：
  slot0=in, slot1=o, slot2=ff1, slot3=ff2
  （對應 l{L}_in_s_x, l{L}_o_s_x, l{L}_ff1_s_x, l{L}_ff2_s_x）
- 矩陣（matrix）更新粒度：以一整個 activation 矩陣為單位（例如 X[列(Row)=node/token][行(Colume)=d]），收完/算完該矩陣後才更新一次，不允許每個向量都更新。

[W-2] 既有權重載入段落保留，但語意改為 seed（只供第一次使用）
- 既有 unified PARAM stream 中的量化參數段落（例如 QUANT_SX_*；FP32 raw words）必須保留，不改順序、不改 dtype、不改 expected_len。
- 新增語意：上述 s_x 僅作為 seed/初始值。
  - 第一次使用某 layer/slot 時：用 seed（或以 seed 初始化 cache）
  - 之後：以動態估測寫回 ScaleTable 的值為準（seed 不再覆蓋）

[W-3] 新增：ScaleTable（SRAM 內的 per-layer table；層數可變）
- 動機：模型層數 N_LAYERS 可能是 runtime cfg；因此 s_x/inv_s_x 必須放在 SRAM 的 table 形式，避免用 compile-time 固定層數的 regs 陣列綁死架構。
- SramMap.h 必須新增（或等價表示）ScaleTable 的配置（u32 word addressing）：
  - SCALE_TABLE_BASE
  - SCALE_TABLE_WORDS
  - SCALE_ENTRY_STRIDE_WORDS（建議 stride 以 ALIGN_WORDS 或 W_LANES 對齊，便於 burst/切片）
  - 規劃建議：ScaleTable 放在 IO_REGION 的子區（語意上屬於 runtime I/O/metadata），不得污染 W_REGION 的 PARAM 區語意。
- 每個 layer L 的 entry（以 u32 words 表示；slot_count=4）至少包含：
  1) sx[L][4]
  2) inv_sx[L][4]
  3) amax_prev[L][4]（平滑用的歷史狀態；見 [W-6]）
  4) valid_mask[L]（1 個 u32，低 4 bits 對應 4 slots；1=已動態更新可用，0=尚未更新、需用 seed）
- 本版不要求 commit flag/version two-phase commit（此議題先不納入本次更新）。

[W-4] 新增：LayerScaleCache（本層快取；避免本輪反覆讀 SRAM）
- 每次進入某 layer L 計算前，Top/Layer 必須先把該 layer 的 ScaleTable entry 載入到暫存器快取：
  - sx_cache[4], inv_sx_cache[4], amax_prev_cache[4], valid_mask_cache
- 該 layer 計算期間（處理其矩陣/向量/tiles 時）：
  - 不允許每次使用尺度都回 SRAM 讀取（本輪只用 cache）
- 多 context overlap 時，cache 的保存不可互相覆寫：
  - 最少需能支援 CTX_ACTIVE 與 CTX_NEXT 的獨立 cache（或以「同一時間只允許一個 context compute」等價方式保證不互撞，但行為需 deterministic）。

[W-5] 新增：Scale Estimator（做法 A；專用 compute-only datapath）
- Scale Estimator 必須是獨立於主計算 datapath 的子模組/子 FSM（English: dedicated estimator datapath）：
  - 僅做比較/乘加/shift 等可合成運算
  - 不得發任何 SRAM request（避免干擾 single-port 1-issue）
- Estimator 輸入：
  - amax_obs（本次矩陣觀測到的最大絕對值）
  - amax_prev_cache（上一輪平滑狀態）
  - inv_sx_cache（用於 freeze/fallback）
- Estimator 輸出：
  - inv_sx_next（本次更新的倒數尺度）
  - sx_next（可選；若需要 dequant/debug；與 inv_sx_next 同步更新）

[W-6] 動態更新的保護措施（Protection；必做）
- Headroom（避免雜訊突然變大導致飽和）：
  - 以常數 margin（shift/add 近似）做保守放大：amax_guard = amax_obs * (1 + margin)
- Smoothing（避免尺度抖動）：
  - 以 shift-based EMA（或等價可合成平滑）更新：
    amax_filt = amax_prev + ((amax_guard - amax_prev) >> SMOOTH_SHIFT)
  - 並寫回 amax_prev_next = amax_filt
- Clamp / Freeze（避免除以 0 或極端值）：
  - 若 amax_filt == 0（或小於 amax_min）→ freeze：inv_sx_next = inv_sx_cache（維持舊值），sx_next 亦維持對應值

[W-7] 倒數（inv_s_x）的計算方法（NR reciprocal；固定迭代次數）
- inv_sx_next 建議以 Newton-Raphson（NR）做 reciprocal 近似，固定迭代次數 K（compile-time 常數；deterministic）：
  - r_{i+1} = r_i * (2 - amax_filt * r_i)
  - inv_sx_next = QGAIN * r_K（QGAIN 為常數，用於對應你的 quant range，例如 int8 的 Qmax=127；具體常數來源需 SSOT）
- sx_next 與 inv_sx_next 必須同一次更新同步寫回 ScaleTable（不可一新一舊）。

[W-8] 更新時序（matrix-level；下一次使用才生效）
- 本次矩陣的量化/運算使用 inv_sx_cache（seed 或舊動態值）。
- 同時旁路累積 amax_obs（abs+max），不得為了算 amax 額外掃 SRAM。
- 在該矩陣處理結束（或可定義的矩陣邊界）後：
  - Estimator 產生 inv_sx_next / sx_next / amax_prev_next
  - 寫回 ScaleTable[L] 對應欄位並更新 valid_mask 的對應 bit
- 下次再進入同一 layer/slot 時，才載入新的值生效（本輪不 retroactively 改變）。

[W-9] TB/驗證一致性（必須可重現）
- 若啟用本條款的動態尺度更新，TB 的 reference/軟體端比對流程必須採用同一套：
  - 同樣的 headroom、smoothing、clamp/freeze、NR 迭代次數與更新時機
- 若 bring-up 階段需要固定尺度以對齊舊 golden：
  - 允許暫時關閉動態更新（只用 seed），但必須在 milestone report 的 Known limitations 明確註記。

【M) v11.8 新增：Input Elastic FIFO（ac_channel）— 必做】
====================================================================

[M-1] 目的（English: decouple input stream）
- 以「小而固定深度」的 FIFO 解耦 data_in 的抖動/背壓，避免上游送資料被卡死。
- Elastic FIFO 不是「完整 payload buffer」：不保證能放下整筆 CFG/PARAM/INFER payload。

[M-2] 實作形式（必遵守）
- Top 內部必須有一個 elastic FIFO（建議命名：in_fifo）：
  - in_fifo 型別：ac_channel<ac_int<32,false>>（或等價 data_word_t）
  - data_in（外部 channel）→ 由 Top 收進 in_fifo → 內部 consumer（CFG_RX/PARAM_RX/INFER_RX）再從 in_fifo 取用
- FIFO 深度：
  - 由 Catapult/HLS 設定（可用 pragma 或工具設定）；咒語不寫死數字
  - 但每次里程碑交付（Mxx_report.md）必須記錄本次使用的 FIFO 深度（方便可重現）

[M-3] 行為約束（避免規格歧義）
- Top 不能因為 internal consumer 尚未開始而拒絕吸收 data_in；只要 in_fifo 未滿，就應盡量吸收 data_in。
- 若 in_fifo 滿：允許對 data_in backpressure（channel rdy 拉低）。
- 所有 length check（EXP_LEN_*）仍以「被 consumer 實際取用的 word 數」為準：
  - 不可用 in_fifo 的暫存量來假裝已完成 payload。
- Gate 建議（可選但強烈建議加嚴）：
  - scripts/check_design_purity.sh 可新增檢查：DESIGN 禁止使用 host-side queue/ STL 容器來當 FIFO（必須用 ac_channel 或等價 HLS-safe FIFO）。

====================================================================
【N) v11.8 新增：IO_REGION（輸入/輸出/回吐）— 建議必做】
====================================================================

[N-1] 動機
- v11.7.2 定義 W_REGION 可包含 input y 等 raw words；功能上可行，但語意容易混淆。
- 本版在「單一主 SRAM」不變的前提下，將 raw-words 區拆成兩個 logical sub-region：
  - W_REGION：僅放 weights/params/bitpack（受 SET_W_BASE + WeightStreamOrder.h 管控）
  - IO_REGION：僅放 input_y staging / output staging / debug dump（可選落地；預設仍以 data_out 串流為主）

[N-2] 規格要求（SramMap.h 必須提供）
- SramMap.h 必須新增（或等價表示）：
  - W_REGION_BASE, W_REGION_WORDS（同 v11.7.2；語意更嚴格：僅權重/參數）
  - IO_REGION_BASE, IO_REGION_WORDS（新增：I/O staging）
- SET_W_BASE 的 range check 只允許落在 W_REGION 範圍內（不得落在 IO_REGION）。

[N-3] input_y / output 落點規則（必遵守）
- 預設：
  - input_y 由 data_in 串流直接被運算消費（不要求落地 SRAM）。
  - 最終輸出（logits/x_pred）預設直接走 data_out（同 v11.7.2 FinalHead 規則）。
- 若需要落地（staging）：
  - input_y staging：只能落地到 IO_REGION（不得混到 W_REGION；避免污染權重區）
  - output staging：只能落地到 IO_REGION（例如 OUT_BUF）；不得放在 SCRATCH

====================================================================
【O) v11.8 新增：Multi-context overlap（CTX0/CTX1 交錯）— 本版納入規格】
====================================================================

[O-0] 名詞
- Context（CTX）：一次 INFER 的「獨立狀態集合」（X pages + 必要的 per-context buffers，例如 KV cache）。
- 本版規格要求最少支援 2 個 context：CTX0、CTX1（English: 2-context interleaving）。

[O-1] 目標（English: accept next INFER early）
- 允許上一筆推論尚未完成（例如後段 FFN/Head/輸出仍在進行）時，
  Top 先接受下一筆 INFER 的 input（接收與 staging），並在資源允許時先做前段運算（例如 Preproc/QKV），提升吞吐。

[O-2] 外部介面不變（Contract 不變）
- 不新增 channel、不新增 opcode；仍使用既有：
  - ctrl_cmd / ctrl_rsp / data_in / data_out（4 channels）
- 不新增 ctx_id 外部欄位；輸出順序固定為 INFER 被 Top 接受的順序（FIFO order）。

[O-3] Top-FSM 行為擴充（覆蓋 v11.7.2 的「*_RX 期間禁止其他 ctrl_cmd」規則；本條款優先）
- 在 v11.8 overlap 模式下，Top 允許：
  - 當 CTX_ACTIVE 正在 COMPUTE（非 *_RX 接收階段）時，
    接受下一筆 ctrl_cmd: INFER（若仍有空的 CTX slot），並進入對應的 INFER_RX(CTX_NEXT) 接收 input。
- 限制：
  - 任一時刻只允許「一個 RX」：也就是同時只收一筆 payload（CFG/PARAM/INFER）；
    但可以在 CTX_ACTIVE COMPUTE 時，收 CTX_NEXT 的 INFER payload。
  - 若兩個 context 都忙（CTX0/CTX1 皆佔用）→ 回 ERR_BUSY。
  - CFG/PARAM 流程仍需在「無 in-flight inference」時執行（避免規格爆炸）：
    - 若有 in-flight inference 時收到 CFG_BEGIN/CFG_COMMIT/LOAD_W/SET_W_BASE → ERR_BUSY。

[O-4] per-context PERSIST buffers（必遵守；這就是你要的「KV 需要雙份」）
- 啟用 multi-context overlap 時，以下資料必須 per-context 各一份（不可共用同一段 SRAM）：
  1) X pages：每個 context 各有 2-page ping-pong
     - X_CTX0_PAGE0 / X_CTX0_PAGE1
     - X_CTX1_PAGE0 / X_CTX1_PAGE1
  2) KV cache（S1）：每個 context 各有 SCR_K / SCR_V
     - SCR_K_CTX0 / SCR_V_CTX0
     - SCR_K_CTX1 / SCR_V_CTX1
- 以上 per-context buffers 雖位於 SCRATCH/ACT region，但其 lifetime_class 視為「CTX_PERSIST」（在該 context 完成前不可覆寫）。

[O-5] SCRATCH 的使用策略（避免 SRAM 爆炸）
- 不強制把所有 scratch 做 double buffer。
- 只對「需要並存」的狀態做 per-context duplication（例如 KV、X pages）。
- 其餘中間量優先 local streaming buffer（見【P】）；
  若不得已落地 SRAM，才允許 per-context duplication，但必須在 SramMap.h 明列並在 report 說明原因。

[O-6] dual-port SRAM 仲裁建議（對應你有 true 2-port）
- 建議：
  - port0：服務 CTX_ACTIVE 的 X/SCR 讀寫（主運算關鍵路徑）
  - port1：服務 CTX_NEXT 的 input staging / KV build / W prefetch（若當下沒有請求可借用給 port0 類請求）
- 仲裁仍需 deterministic（同 v11.7.2【8.5】精神）。

====================================================================
【P) v11.8 補強：SCRATCH / Local buffering 規則（禁止錯用 SRAM）】
====================================================================

[P-1] softmax partial 禁止落地 SRAM（必遵守）
- softmax 的 max/sumexp/denom/reciprocal、以及單一 q_node 的 acc_vec 等「小中間量」：
  - 必須使用 local regs 或小型 HLS-safe RAM（例如固定深度 FIFO/array），不得佔用 SRAM pages。
  - 理由：這些量的 liveness 極短，落地 SRAM 只會浪費頻寬與容量，且增加仲裁不確定性。

[P-2] FFN1 / Attn_out 優先 streaming（強烈建議）
- 若後段 consumer（例如 W2 / residual / LN）能 streaming consume：
  - FFN1 hidden、Attn_out 不應整頁落地 SRAM。
  - 允許使用固定深度的小 FIFO / line buffer（可用 ac_channel 或等價 HLS buffer）。
- 若因排程/資源隔離需要落地 SRAM：
  - 僅允許落地到 SCRATCH（或 per-context scratch），且必須在 SramMap.h 宣告，並在 report 說明為何不能 streaming。

====================================================================
【Q) v11.8 補強：TB/驗證規則（overlap 模式下的可驗證性）】
====================================================================

[Q-1] 輸出順序（必遵守）
- 無論是否 overlap，data_out 的正常輸出順序固定為：
  - 依 INFER 被 Top 接受（回 RSP_OK(INFER)）的先後順序輸出 logits/x_pred。
- 不新增 ctx_id 於外部介面；TB 以「命令發送順序」對應輸出即可。

[Q-2] TB 可選擴充（建議）
- TB 可在 overlap 模式下做 stress test：
  - 在上一筆 INFER 仍在運算/輸出期間，提前送下一筆 INFER + input_y
  - 驗證 Top 是否能正確處理、且輸出順序無誤
- 若未做 overlap stress test，功能仍可 PASS，但在 report 的 Known limitations 必須註明。

====================================================================
【R) v11.8 補強：離線生成器（M16）需要新增的輸出欄位】
====================================================================

[R-1] SramMap.h 需新增（或擴充）欄位
- region: X_CTX0_PAGE0 / X_CTX0_PAGE1 / X_CTX1_PAGE0 / X_CTX1_PAGE1 / SCR_K_CTX0 / ... / W_REGION / IO_REGION
- lifetime_class: PINGPONG / SCRATCH / CTX_PERSIST / PERSIST
- ctx_id: 0/1（若不適用填 0）
- alias_group: 僅允許對 SCRATCH 類使用；CTX_PERSIST 不可 alias（直到該 context 完成）

【F) v11.7.1 新增：Global Hard Rules（全域硬規則；必遵守）】
====================================================================

[F-1] DESIGN vs TB boundary（設計/測試邊界；零容忍）
- DESIGN（src/、include/、design/ 等任何會被 HLS analyze/synth 的檔）：
  - 禁止 include 任何 trace headers：例如 "*_step0.h"、"layer*_*.h" 這類 checkpoint trace
  - 禁止使用 trace tensor_shape / trace tensor 內容當作設計行為的依據
  - 即使包在 #ifdef/#ifndef 也不允許（原因：會讓設計受 TB 宏控制，且打包時常因缺檔而分析失敗）

- TB（tb/）：
  - 可以 include trace（step0 headers），用於比對與 debug（READ_MEM + compare）
  - trace 永不進 git；交付壓縮檔可帶最小 end-to-end trace（見 v11.7 原文）

[F-2] 禁 trace bypass（禁止用抄 golden 讓 end2end 假通過）
- 禁止任何形式：
  - 先從 input_y 找 sample_idx
  - 然後把 output_logits_step0 / output_x_pred_step0 直接寫到 SRAM / data_out
- logits/x_pred 必須由設計運算得到；TB 只能做載入與比對。

[F-3] 可合成安全子集合（HLS-safe subset；以 gate 強制）
- DESIGN 禁止：
  - <cmath> 的 sqrt/exp/log/pow 等 host math
  - union / memcpy 這類 bit-cast 技巧（若要 bits<->float，使用既定的 raw-fp32 轉換 helper，且需符合 CodeGen Guide）
  - dynamic allocation / exceptions / RTTI / iostream / fstream / heavy STL
- DESIGN 型別優先：
  - ac_int / ac_fixed / ac_ieee_float<binary32>（依 QuantDesc/SoftmaxApprox/LayerNorm 規範）
- 任何違反上述項目，都必須讓 scripts/check_ccs_style.sh 直接 FAIL。

[F-4] SSOT（Single Source of Truth）強制收斂
- shape / expected_len / SRAM base&size / PARAM offsets 的唯一來源：
  - ModelDesc.h / ModelShapes.h / SramMap.h / WeightStreamOrder.h
  - 若存在 gen/ 版本（M16 pipeline），Top/TB/blocks 一律 include gen/ 版本；Bringup 版只保留 legacy，不可用於正式編譯。
- 禁止 Top/TB/blocks 各自 hardcode base/len/offset；否則 regression 會出現「寫A讀B」的假錯誤。

[F-5] Gate（你必須讓規則「機械化」）
- scripts/check_ccs_style.sh 必須新增檢查（DESIGN 範圍）：
  - FAIL if found: #include ".*step0.h"
  - FAIL if found: trace_.*(tensor_shape|tensor)
  - FAIL if found: AECCT_.*_TRACE_MODE（或任何 trace bypass 宏）

- 本規格的「bank」在語意上等同於「logical region」（English: logical regions / address ranges）：
  - X_PAGE0 region：只存 X ping-pong 第 0 頁
  - X_PAGE1 region：只存 X ping-pong 第 1 頁
  - SCRATCH region：只存 scratch（含你選的 S1：KV Cache）
  - W_REGION region：只存參數/資料的 raw words（weights / bit-packed / bias / input y 等）
- v11.5（你選的方案1 / M2-2P）允許「物理上」用 *單一* true 2-port SRAM macro 承載上述全部 regions：
  - 外部仍以 u32 word addressable（addr_word）定義；READ_MEM 只依 SramMap.h 的 base/size decode 落點屬於哪個 region。
  - region 的容量可以極度不平衡；SramMap.h 的離線生成器必須允許你按模型需求配置各 region 的 size（English: region sizes are configurable; not equal-split）。
- 禁止：
  - 把權重「依種類平均分散到不同 region」、或把單一線性位址「用最低位元交錯」做全域 interleave（這會破壞串流/對齊與 TB 一致性）。
- W_REGION 的內部讀取仍允許寬讀口/向量化（wide read / burst gather）：
  - 介面不變：TB↔Top 的 data_in/data_out 一律維持 u32 raw word（不改【6)】）。
  - 若物理 SRAM 支援 256b（或更寬）讀：可直接對應 W_LANES=8 一拍讀出。
  - 若物理 SRAM 只有 32b：Top 必須以「連續讀 W_LANES 次」做 burst gather，再放入 block-local FIFO/line-buffer 做重用（功能等價；吞吐視排程而定）。

【1) Profile（避免規格互相覆蓋）】
- PROFILE_RUNTIME（預設，對齊 ）：
 - runtime CFG 可變：CODE_N、CODE_C、N_LAYERS、D_MODEL、D_FFN、(可選) D_LPE
 - N_HEADS 固定為 8；必須檢查 cfg_d_model % 8 == 0
 - 因硬體陣列需固定：定義 compile-time ModelLimits（MAX_*），runtime cfg 只能在有效範圍（<=MAX_*）內運作
 - TB 的「模型架構設定」會先讀 ModelDesc.h 當作 default，再用 CFG 流程寫進 Top regs
- PROFILE_FIXED（對齊 ，用於快速 bring-up / trace 對齊）：
 - 所有 shape compile-time 固定；CFG 只驗證 Top-FSM 流程，不改變陣列大小與資源配置

【2) 總體架構與分塊（Block partition：5 大 block）】
(1) Top：仲裁者/控制器/共享 SRAM 擁有者 + FSM + runtime cfg regs + 各段 base/len 記錄 regs
(2) PreprocEmbedSPE：embed + SPE；LPE 不做額外計算（LOAD_W 直接載入/或專段載入）
 - 方案B：輸出一律寫入 X_out（由 Top 指定頁），完成後由 Top swap bank
(3) TransformerLayer：單一層硬體，time-mux 重複執行 N_LAYERS 次
 - 內部可拆 private functions，但維持單一對外介面
 - 必須包含：Q/K/V 投影、多頭 attention、concat、out projection、residual、FFN(兩層 linear + ReLU)、以及該層的 norm（依 notebook 流程）
 - 方案B（計算結果暫存）：Top 以 base/len 方式傳入 X_in/X_out 與 SCRATCH 的 base（避免子模塊自行分配/計算位址）；layer 完成後由 Top 進行 bank swap
(4) LayerNormBlock：獨立 LN，供 Top 插入 mid LN / end LN（runtime cfg 時用 MAX_D_MODEL buffer + cfg_d_model 有效長度）
 - 方案B：以兩-pass 規則運作（pass1 只讀 X_in；pass2 讀 X_in 寫 X_out），base 由 Top 指定
(5) FinalHead：Transformer 後額外兩層 FFN + logits 計算 + x_pred/sign(logits*y) 翻轉邏輯
 - 支援 OUTMODE：輸出 x_pred / logits / none（由 ctrl SET_OUTMODE 控制；none 用於 debug/halt）
 - 方案B：輸入從最新的 X_in 頁讀取；中間暫存僅可使用 SCRATCH；最終結果預設直接走 output channel（若要落地 SRAM，必須在 SramMap.h 明確分配 PERSIST/OUT_BUF）

【3) LayerNorm 插入規則】
- mid LN 插入層 index：mid_index = (N_LAYERS//2) - 1
 - N_LAYERS=5 時：mid_index=1，表示 layer 0、1 做完後插入一次 LN（對齊 notebook 的 encoder_norm_mid_out 概念）
 - runtime cfg 太小導致 mid_index<0：可不插 mid LN（需明確邊界處理）
- Transformer 全部完成後一定要做 end LN（encoder_norm_end），再進 FinalHead

【4) 介面：data_in/data_out 共用 + ctrl_cmd/ctrl_rsp（模式分離 + 小位元數）】
- 四條 channel（建議命名固定，避免你之後一直改 TB）：
 - ctrl_cmd : ac_channel<ac_int<16,false>> // 小位元數；主要放 opcode
 - ctrl_rsp : ac_channel<ac_int<16,false>> // 小位元數；回報 OK/DONE/ERR
 - data_in : ac_channel<ac_int<32,false>> // 統一 u32 raw words（CFG/bias/weights/input y/command args）
 - data_out : ac_channel<ac_int<32,false>> // 統一 u32 raw words（x_pred/logits/debug readback）

- 「小 ctrl，大資料都走 data」的規則（你選 2B + 2C 的精神）：
 - ctrl_cmd 只放 opcode（+少量 flags/小 arg，如果你之後真的需要）
 - 任何需要參數的命令，一律在 ctrl_cmd 之後，立刻從 data_in 再讀固定個數的 u32 arg words
 - 這樣就不必把 addr/len/outmode…硬塞進 ctrl 的 bitfield，ctrl 位元數可以很小
 - 不需要 ST_BUSY：
 - 若 Top 忙到不能收新命令：要嘛 ctrl_cmd 的寫入端會被 back-pressure（channel 本身的 rdy）
 - 要嘛回 RSP_ERR(ERR_BUSY)

- 模式分離（Top-FSM 解析 data_in 的方式）：
 - IDLE：只接受 ctrl_cmd（以及該 cmd 的 args）
 - CFG_RX：data_in 只允許 cfg words（按 ModelDesc.h 欄位順序；全 u32）
 - BIAS_RX：data_in 只允許 bias words（全 u32 raw）
 - WEIGHT_RX：data_in 只允許 weight words（FP32 raw + BITPACK raw）
 - INFER_RX：data_in 只允許 input y words（FP32 raw bits，u32）
 - HALTED：僅在 D1 斷點命中時進入；freeze 住運算與 SRAM 寫入；只允許 READ_MEM / DEBUG_CFG(RESUME/CLEAR) / SOFT_RESET

- 命令與 payload 的邊界規則（避免 TB/硬體解析歧義）：
 - 進入某個 *_RX 模式的 cmd（CFG_BEGIN/LOAD_W/INFER；LOAD_BIAS 為 [legacy]）只能在 IDLE 下發
 - *_RX 模式期間禁止下其他 ctrl_cmd（除非你之後特別設計可中斷），結束後 Top 會回 RSP_DONE 再回到 IDLE

【5) Ctrl 命令集（ctrl_cmd/ctrl_rsp，小位元數；大參數走 data_in）】
- ctrl_cmd 格式（建議 16-bit）：
 - [7:0] opcode
 - [15:8] reserved（目前填 0；未來要加 flags 再用）

- ctrl_rsp 格式（建議 16-bit）：
 - [3:0] rsp_kind：0=OK，1=DONE，2=ERR
 - [7:4] reserved
 - [15:8] payload（對 OK/DONE：回傳對應的 opcode；對 ERR：回傳 err_code）

- opcode 定義（目前用不到大 arg，所以 opcode 8-bit 很夠）：
 - 0x01 CFG_BEGIN
 - 0x02 CFG_COMMIT
 - 0x04 LOAD_W（主路徑：LOAD_PARAM；進入 PARAM_RX）
 - 0x05 SET_OUTMODE （需要 1 個 u32 arg，走 data_in）
 - 0x06 INFER
 - 0x07 READ_MEM （需要 2 個 u32 arg，走 data_in；debug/回吐用）
 - 0x08 DEBUG_CFG （需要 1 個 u32 arg，走 data_in；選 checkpoint/是否 dump）
 - 0x09 SET_W_BASE（需要 1 個 u32 arg，走 data_in；設定 param_base_word）
 - 0x03 LOAD_BIAS [legacy]（分離式 bias 流程；相容用）
 - 0x7F SOFT_RESET （可選：只 reset FSM/flags，不清 SRAM；bring-up 很好用）

- 各命令的 args（都用 u32）：
 - SET_OUTMODE：arg0 = outmode
 - 0：OUTMODE_XPRED
 - 1：OUTMODE_LOGITS
 - 2：OUTMODE_NONE（不吐任何正常推論輸出；保留 data_out 給 D1 meta / READ_MEM；debug halt 時建議用這個）
 - READ_MEM：arg0 = addr_word（SRAM 的「word address」，以 u32 為單位）
 arg1 = len_words（要讀多少個 u32 words）
 - Top 收到後：依序把 len_words 個 u32 丟到 data_out，最後回 RSP_DONE(READ_MEM)
 - 這條就是你提的「直接做一個能讀 SRAM 的 ctrl」：debug/回吐一律走這條，不必再做第二條 debug channel
 - DEBUG_CFG：arg0 = packed（你已選用 Debug 策略 D1：就地 halt + READ_MEM；不需要 PERSIST_DEBUG）
 - arg0 layout（u32）：
 - [0] dbg_en（1=啟用；0=關閉）
 - [1] dbg_halt_en（1=命中 checkpoint 時「就地停住」；0=不 halt）
 - [3:2] dbg_action（0=SET/ARM：套用本次設定；1=RESUME：從 HALTED 繼續；2=CLEAR：清除斷點；3=reserved）
 - [11:4] dbg_kind（enum；例如 1=X_IN，2=K_CACHE，3=V_CACHE，4=ATTN_OUT，5=MID_LN_OUT，6=END_LN_OUT…）
 - [19:12] dbg_layer_id（0..N_LAYERS-1；若不適用填 0）
 - [27:20] dbg_head_id（若不適用填 0）
 - [31:28] reserved
 - D1（你選的策略；推薦、最省 SRAM）：
 - DEBUG_CFG(SET/ARM) 只是在 Top 設定「斷點條件」（kind/layer/head + halt_en）
 - INFER 執行期間若命中 checkpoint：
 1) Top 進入 HALTED（freeze）：停止往下運算、停止任何 SRAM 寫入（確保觀測資料不被覆寫）
 2) 回 ctrl_rsp：RSP_ERR(ERR_DBG_HALT)（這不是致命錯誤；表示「依 debug 設定停住」）
 3) 立即在 data_out 送 2 個 u32 meta words（不混在 logits/x_pred 中）：
 - meta0 = addr_word（要觀測資料所在 SRAM word address）
 - meta1 = len_words（要觀測資料長度，以 u32 words）
 - 回應順序釘死：必須先寫 ctrl_rsp（RSP_ERR(ERR_DBG_HALT)），之後才允許在 data_out 推 meta0、meta1（且 meta0 必先於 meta1）。
 - 注意：雖然 ctrl_rsp 與 data_out 是不同 channel、理論可並行，但為了 TB/host 不踩死鎖，這裡規格要求固定順序。
 - Top 不得因 data_out backpressure 延遲 ctrl_rsp；meta0/1 必須可先暫存在內部 2-word buffer，待 data_out ready 再送出。

 4) TB 可用 READ_MEM(meta0, meta1) 讀回資料做比對/畫圖
 5) TB 若要繼續同一次 INFER：送 DEBUG_CFG(RESUME)（dbg_action=RESUME）→ Top 離開 HALTED 繼續跑
 - 注意：HALTED 狀態下允許的 ctrl_cmd（其餘視為 ERR_BAD_STATE）：
 - READ_MEM（讀回當下 SRAM 內容）
 - DEBUG_CFG（RESUME / CLEAR）
 - SOFT_RESET（放棄本次推論並回到 IDLE）
 - D0（legacy，可保留但你目前不採用）：copy-to-PERSIST_DEBUG
 - 若你未來想做「不中斷推論、把 checkpoint copy 到固定區域」才需要
 - 這條路會需要在 SramMap.h 分配 PERSIST_DEBUG，並在命中點做 copy，再由 TB READ_MEM(PERSIST_DEBUG) 讀回

- Top-FSM 的基本行為（回應順序）：
 - cmd 被接受時：先回 RSP_OK(opcode)
 - 需要等 payload/運算完成的 cmd（LOAD_W/INFER/READ_MEM；LOAD_BIAS 為 [legacy]）：完成時再回 RSP_DONE(opcode)
 - 若發生錯誤：回 RSP_ERR(err_code)
 - 一般錯誤：回到 IDLE（必要時也可保留現有載入進度，依你喜好）
 - 例外：ERR_DBG_HALT（D1 斷點命中）→ 進入 HALTED，不回 IDLE；只允許 READ_MEM / DEBUG_CFG(RESUME/CLEAR) / SOFT_RESET

- error code 建議枚舉（你說要我先給選項，你再多選）：
 - 0x01 ERR_BUSY // 非 IDLE 狀態下收到 cmd
 - 0x02 ERR_BAD_STATE // 指令順序不合法（沒 CFG_COMMIT 就 LOAD_W / INFER…）
 - 0x03 ERR_CFG_LEN_MISMATCH // CFG words 數量不符
 - 0x04 ERR_BIAS_LEN_MISMATCH // bias expected_len 不符
 - 0x05 ERR_W_LEN_MISMATCH // weights expected_len 不符
 - 0x06 ERR_BITPACK_PAD // bitpack 最後 word padding/valid_bits 檢查失敗
 - 0x07 ERR_BAD_OUTMODE // SET_OUTMODE 參數非法
 - 0x08 ERR_MEM_RANGE // READ_MEM addr/len 超界或跨 region 禁止
 - 0x09 ERR_CFG_ILLEGAL // cfg 自洽性檢查失敗（例如 d_model%N_HEADS!=0、超 MAX…）
 - 0x0A ERR_TIMEOUT_INTERNAL // 子模塊卡住（debug 用，可選）
 - 0x0B ERR_DBG_HALT // 依 DEBUG_CFG 斷點命中而 halt（D1）；非致命；Top 留在 HALTED
 - 0x0C ERR_PARAM_LEN_MISMATCH // PARAM expected_len 不符（主路徑）
 - 0x0D ERR_PARAM_BASE_ALIGN // SET_W_BASE 對齊檢查失敗（可選）
 - 0x0E ERR_PARAM_BASE_RANGE // SET_W_BASE 範圍檢查失敗（主路徑）

-（回答你問的「什麼情況需要很大的 arg？」）
 - 只要你想在 ctrl 內一次塞進「addr/len/layer/head/checkpoint/mode…」這種東西，就會爆
 - 例如 READ_MEM：addr(>=20~24 bits) + len(>=16 bits) + mem_sel + flags
 - 例如 debug dump：layer_id + head_id + tile_id + kind + stride…
 - 所以我才把「大參數」全部移到 data_in 的 u32 args，ctrl 只留 opcode → 最省位元又最彈性

【6) data_in/data_out：字格式（你已選 bit-packed；全部走 u32 raw word）】
- 統一用「raw word」串流：DATA_W = 32，1 word = 1 個 u32
- FP32 類資料（weights / bias / input y / x_pred / logits）：
 - TB 端把 double/float 轉成 fp32
 - 再把 fp32 以 IEEE754 bit-pattern 當作 u32 raw word 傳入/傳出
 - Top 端 SRAM 也以 raw u32 存；使用時再轉回 ac_ieee_float<binary32>

- BITPACK 類資料（例如校驗矩陣 H、Mask）：
 - TB 端把 bitstream packing 成 u32 raw words
 - bitstream 的第 0 個 bit → 放在 word[0] 的 LSB（bit0）
 - 也就是：bit_idx = 0 對應 (word_idx=0, bit_in_word=0)
 - 最後一個 word 若沒擺滿：剩餘 bits 全置 0
 - Top/子模塊讀取邊界必須用 valid_bits（只對 BITPACK 段需要；FP32 段不需要）

- 驗證與 expected_len（禁止 TODO；以 header 為準）：
 - include/ModelDesc.h：CFG_RX 欄位順序/數量（EXP_LEN_CFG_WORDS）
 - include/ModelShapes.h：INFER_RX 與正常 OUT 的 expected_len（EXP_LEN_INFER_IN_WORDS / EXP_LEN_OUT_*），以及 words_fp32/words_bits 的計算規則
 - include/WeightStreamOrder.h：定義 LOAD_BIAS/LOAD_W 的 exact order + 每段 len(words)/dtype/shape + SRAM offset（FP32/BITPACK）；並提供 EXP_LEN_BIAS_WORDS / EXP_LEN_W_WORDS
 - （可選/legacy）include/BiasDesc.h（或 bias_*.h）：若你把 bias 串流獨立出來可用；但 推薦以 WeightStreamOrder.h 內的 bias meta / EXP_LEN_BIAS_WORDS 為準
 - Top 在 BIAS_RX/WEIGHT_RX：逐段驗證 expected_len（少送/多送/長度不符 → ERR_*_LEN_MISMATCH）
 - BITPACK 段額外驗證：最後一段 word 的 padding bits 必須為 0（若你選擇做這個檢查）
-（v11 補充：配合 W bank 寬讀口）W_REGION 的所有段落 base/len 必須對齊到 W_LANES（不足以 padding=0 補齊；padding words 仍算在 expected_len 內，並由 WeightStreamOrder.h 宣告）。

【7) 型別規格（非常重要）】
- 全部使用 AC datatypes：ac_int / ac_fixed / ac_ieee_float<binary32> / ac_channel（必要時可用 ac_shared/ac_sync）
- data channel 建議型別：
 - data_word_t = ac_int<DATA_W,false>（預設 DATA_W=32）
- weights.h：
 - 浮點權重以 double 儲存（TB 端轉 fp32 再以 raw bits 送入）
 - bit-packed 權重以 packed words 儲存（TB 端直接送 raw words）
- Top SRAM：
 - 權重/bit-packed/輸入 y/bias 皆可用「raw-word SRAM」集中存放（W_SRAM）
 - activation 中間結果可用 fp32 SRAM（ACT_SRAM）或同一塊 SRAM 分區（但務必清楚切分）
- 避免 using type alias（除非我明確允許）

【8) SRAM：集中在 Top、inner-most contiguous、方案B（X ping-pong + scratch reuse；v11.5 實作：單一 2-port SRAM + logical regions）】
- 權重集中在 Top SRAM（v11.5：建議用「單一 true 2-port SRAM macro」承載所有資料，再以 SramMap.h 切成 logical regions）
- tensor 線性化採 inner-most dimension contiguous（d/行(Colume) 連續）
- 方案B 定義：主幹 activation「X」使用 2-page ping-pong；其餘中間結果一律視為 scratch（可重用/可覆寫），由 Top 排程保證不覆寫仍需資料
-（v11 固定 logical regions；English: fixed logical regions by contention class）：
 - X_PAGE0 region：只放 X_PAGE0
 - X_PAGE1 region：只放 X_PAGE1
 - SCRATCH region：只放 SCRATCH（含 S1：KV cache）
 - W_REGION region：只放 W_REGION（weights / bit-packed / bias / input y 等；集中連續位址；禁止 interleave）


【8.1) ACT_SRAM 版面（logical regions；實際 base/size 由 SramMap.h 離線生成）】
-（v11 對應關係）：
 - W_REGION → W bank（集中連續位址）
 - X_PAGE0 → X_PAGE0 bank
 - X_PAGE1 → X_PAGE1 bank
 - SCRATCH → SCRATCH bank
A) W_REGION（raw words）：weights / bit-packed / bias / input y 等（依【6)】與【11)】規則）
B) ACT_REGION（raw words，內容視為 fp32 raw-bit pattern）：
 - X_PAGE0：X ping-pong 第 0 頁
 - X_PAGE1：X ping-pong 第 1 頁
 - SCRATCH：各子模塊的暫存區（Q/K/V、attention partial、FFN hidden、residual 暫存等）
 - （legacy，可選）PERSIST_DEBUG：僅供 D0(copy-to-PERSIST) 使用；你目前採 D1 就地 halt，預設不分配

【8.1.1) v11：W bank 寬讀口（wide read port；只針對 W bank；W_LANES=8）】
- 目的：降低權重讀取瓶頸；配合 block 內快取（見【8.4.2】）。
- 外部介面不變：TB↔Top 仍是一個 u32 raw word 串流；Top 內部可用寬 word 存取 W bank。
- 定義：W_LANES = 每個寬 word 包含的 u32 lanes 數（compile-time 常數；以 ModelShapes.h 的 W_LANES 為準；此模型固定為 8，等價 8×32=256-bit；Top/Blocks/TB 共用）。
- 對齊規則（避免 read-modify-write）：
 - W bank 的 base/len、以及 WeightStreamOrder.h 宣告的每段 offset/len，都必須 align 到 W_LANES。
 - 不足的尾端以 padding=0 補齊，padding words 仍算在 expected_len（由 WeightStreamOrder.h 宣告）。

- v11.1 額外硬規則（避免 wide-read 失效或引入複雜的 unaligned gather）：
 - SramMap.h 的 W_REGION_BASE 與 W_REGION_WORDS 必須是 8 的倍數（以 u32 words 為單位）。
 - WeightStreamOrder.h 中「落在 W_REGION」的每一段：
  * offset_w 必須是 8 的倍數（u32 word address 對齊 8-word boundary）。
  * len_w 必須是 8 的倍數；若真實 payload 長度不足 8 的倍數，必須在串流尾端補 0（padding=0）。
  * padding words 必須計入 expected_len（也就是 TB 真的要送出那些 0）。
 - padding 的值（TB→Top）一律送 u32=0x00000000（對 fp32 就是 +0.0 的 bit pattern；對 bitpack 就是全 0 bits）。
- READ_MEM 相容性（debug 友善）：
 - READ_MEM 的 addr_word/len_words 仍以「u32 words」定義，不因內部 256-bit 寬讀口而改變。
 - 若 addr_word 或 len_words 不是 8 對齊：允許 Top 以「先對齊讀出包含區間的寬 word，再做 lane slice」的方式回吐（可較慢但必須正確）。


【8.2) X ping-pong 的頁大小 + 「consumer 控制 swap」規則（支援 blocks 級 pipeline）】
- X 的語意：X[列(Row)=node/token][行(Colume)=d_model]
- 以 fp32 raw words 存放：1 element = 1 word（DATA_W=32）
- 頁大小（words）：
 - X_WORDS = cfg_n_nodes * cfg_d_model
 - X_PAGE_WORDS = align_up(X_WORDS, ALIGN_WORDS) // ALIGN_WORDS 以 SramMap.h 的定義為準（例如 16 words）

- ping-pong 的「頁」定義：
 - page_id = 0/1 對應 X_PAGE0 / X_PAGE1
 - 任何 stage 都必須遵守：只讀 X_in(page_in)，只寫 X_out(page_out)，且 page_out != page_in

- consumer 控制 swap（你指定的規則）：
 - producer（上游 stage）「完成寫入 page_out」之後，只做一件事：把 page_out 以 token/狀態 publish 給下游
 - 真正的「bank_sel 切換」發生在 consumer（下游 stage）開始使用新頁的那一刻
 - 直覺版：資料被下游「開始讀」了，才算 swap 成功
 - 為了避免被覆寫：
 - page 需要有「in_use」（被讀者占用）概念（refcnt 或簡化的 busy flag 皆可）
 - producer 只有在 page_out 沒人占用時才允許覆寫（refcnt==0）
 - in_use 釘死採用 per-consumer bitmask（每個 consumer 對應 1 bit；English: per-consumer in_use mask）：
 - acquire_read(page_id, consumer_id)：in_use_mask[page_id][consumer_id] = 1
 - release_read(page_id, consumer_id)：in_use_mask[page_id][consumer_id] = 0
 - producer 覆寫條件：in_use_mask[page_out] == 0（等價於 legacy 的 refcnt==0；若需要 refcnt，可用 popcount(mask)）
 - 同 cycle 多個 acquire/release：用 set_mask/clear_mask 做原子更新（atomic update）：mask_next = (mask | set_mask) & ~clear_mask

（帶來的好處）
- blocks 級 pipeline 時，上游可以先把結果寫到另一頁並 publish，下游什麼時候接手由下游自己決定
- 也避免你之前擔心的：producer 一寫完就 swap，結果下游其實還沒讀完舊頁

【8.3) 何時 swap bank（用「token/頁狀態」描述；對齊 TB/trace）】
- 在你要的 blocks 級 pipeline 版本裡，「swap」不是硬做在 producer 結尾，而是以下事件：
 - consumer 取得新頁 token / 或 consumer 開始讀新頁（兩者等價）

- stage 串接的 publish/consume（語意版）：
 - PreprocEmbedSPE：produce X_out(page_out) → publish
 - TransformerLayer(layer_id)：consume X_in(page_in) → produce X_out(page_out) → publish
 - mid LN（若啟用）：consume → produce → publish
 - end LN：consume → produce → publish
 - FinalHead：consume 最新 X_in，輸出 logits/x_pred 走 data_out（不再 produce 新 X page）

- 若你先用「單一 Top-FSM、逐段呼叫子模塊」bring-up：
 - 你仍然可以用同樣語意對齊：Top 不在 producer 結尾立刻 flip bank_sel
 - 而是在「準備呼叫下一段 consumer」之前，才把 consumer 的 X_in 指到上一段 publish 的 page
 - 這樣在行為上仍符合「consumer 控制 swap」

【8.4) SCRATCH 的使用規範（同層內有效；跨層可覆寫）】
- SCRATCH 的資料生命週期（liveness）：僅限「單一子模塊/單一 layer run / 單一 LN run」內有效
- 允許的 scratch 類型（可依實作取捨，但必須在 SramMap.h 宣告其 base/size，且可支援 alias/overlay）：
 - SCR_Q / SCR_K / SCR_V（若你選擇先 materialize 再算 attention）
 - SCR_ATT_OUT（sublayer_out 暫存，用於 residual add）
 - SCR_FFN1（FFN 第一層 hidden，用於 ReLU + 第二層）
 - 其他：softmax partial、tile psum、reduce 用累加器等（若需）

【8.4.1) SCRATCH 方案：S1「KV Cache」（你已選；推薦 bring-up）】
- 目標：只在 SCRATCH 落地 K/V（各一份），Q 不落地（on-the-fly），以最小 scratch 換取可控 schedule 與良好 debug 可見性。
- SCRATCH 分割（以 u32 words 為單位；base 由 Top/SramMap.h 提供）：
 - X_WORDS = cfg_n_nodes * cfg_d_model
 - X_PAGE_WORDS = align_up(X_WORDS, ALIGN_WORDS) // 同【8.2】規則
 - SCR_K：shape = [cfg_n_nodes, cfg_d_model]（列(Row)=node、行(Colume)=d_model）
 - base = scratch_base + 0 * X_PAGE_WORDS
 - len = X_PAGE_WORDS
 - SCR_V：shape = [cfg_n_nodes, cfg_d_model]（列(Row)=node、行(Colume)=d_model）
 - base = scratch_base + 1 * X_PAGE_WORDS
 - len = X_PAGE_WORDS
 - 因此：SCRATCH_WORDS（S1）= 2 * X_PAGE_WORDS
 - 註：softmax 的 max/sumexp、單一 node 的輸出向量 y[0..D_MODEL-1] 建議用 reg（不占 scratch）。

- TransformerLayer（單層、time-mux）的內部排程（看得出流水線的版本）：
 Phase A) KV_MAT（一次掃過 X_in；materialize K/V）
 - 讀 X_in(page_in) →（依 notebook/你的 LN 插入規則，可在此做 LN1）→ 計算 K,V → 寫入 SCR_K/SCR_V
 - K/V 在整個 attention 運算期間保持不變（不可被覆寫）

 Phase B) ATTN（Q on-the-fly + 兩-pass softmax；寫出 attention+residual 結果）
 - 對每個 q_node（0..cfg_n_nodes-1）：
 1) 由 X_in(q_node)（或 LN1(X_in)）計算 q_vec（不落地）
 2) pass1：掃 t_node=0..cfg_n_nodes-1
 - score = dot(q_vec, SCR_K[t_node]) + (可選)LPE(q,t) + mask
 - 求 max_score
 3) pass2：再掃 t_node
 - w = exp(score - max_score)
 - sumexp += w
 - acc_vec += w * SCR_V[t_node] // acc_vec 是 D_MODEL 長度，建議用 reg 累加
 4) out_vec = acc_vec / sumexp
 5) out_proj = out_vec * Wo + bo
 6) residual：X_mid = out_proj + X_in(q_node)
 7) 寫入 X_out(page_out) 的對應 node（此時 X_out 暫存的是本層「attention sublayer 後」的結果）

 Softmax 可合成實作（方案A：LUT exp + reciprocal；禁止 std::exp 與除法；English: LUT exp + reciprocal, no division）
 - 定義：x = score - max_score（理論上 <=0），先 clamp：x_clamp = clamp(x, -SOFTMAX_NEG_T, 0)
 - exp_lut：用 compile-time const ROM LUT 查 exp(x_clamp) → w（可選 linear interpolation；LUT index 由 x_clamp 線性映射）
 - sumexp += w；sumexp 最小值需 clamp 到 SOFTMAX_EPS（避免 reciprocal 爆掉）
 - inv_sumexp = rcp_lut(sumexp_clamp)（reciprocal LUT；可選做 1 次 Newton refinement，但仍禁止直接用 '/'）
 - out_vec = acc_vec * inv_sumexp（用乘法取代除法）
 - 需釘死的 compile-time 參數（建議放 ModelLimits.h 或 SoftmaxApprox.h）：
 - SOFTMAX_NEG_T、SOFTMAX_EXP_LUT_SIZE、SOFTMAX_RCP_LUT_SIZE、SOFTMAX_EPS
 - 放置位置釘死（English: header source of truth）：
 - 以上 4 個參數 + exp_lut/rcp_lut 內容 **必須** 定義在 include/SoftmaxApprox.h（Top/TB/Blocks 全部 include 同一份）
 - legacy： 的「ModelLimits.h 或 SoftmaxApprox.h」僅保留文字，不再允許二選一
 - LUT 索引映射規則（避免各處不一致；English: fixed index mapping）：
 - exp_lut：idx = sat_u( (-x_clamp) * SOFTMAX_EXP_IDX_SCALE )；其中 SOFTMAX_EXP_IDX_SCALE=(SOFTMAX_EXP_LUT_SIZE-1)/SOFTMAX_NEG_T，並在 SoftmaxApprox.h 以預先計算常數給出（mapping logic 不使用 '/'）；x=0→idx=0，x=-T→idx=LAST
 - rcp_lut：sumexp_safe = clamp(sumexp, 1, SOFTMAX_SUMEXP_MAX)
 idx = sat_u( (sumexp_safe - 1) * SOFTMAX_RCP_IDX_SCALE )；其中 SOFTMAX_RCP_IDX_SCALE=(SOFTMAX_RCP_LUT_SIZE-1)/(SOFTMAX_SUMEXP_MAX-1)，並在 SoftmaxApprox.h 以預先計算常數給出（mapping logic 不使用 '/'）
 - SOFTMAX_SUMEXP_MAX：必須與 SoftmaxApproxLutData.h 生成參數一致（目前 LUT 以 256 生成；即使 N_NODES 較小也可維持 256 以簡化與保持一致）
 - 推薦 bring-up 初值（可改但必須集中在 SoftmaxApprox.h；English: suggested defaults）：
 - SOFTMAX_NEG_T=10（或 12）；SOFTMAX_EXP_LUT_SIZE=256；SOFTMAX_RCP_LUT_SIZE=256；SOFTMAX_EPS=1e-8（fp32）/ 1 LSB（fixed）
 - 型態建議：LUT/acc/sumexp 可用 ac_fixed（推薦、資源可控）或 ac_ieee_float<binary32>（bring-up）；但不允許呼叫不可合成的 libm exp/div

 Phase C) FFN（在 X_out(page_out) 上做「per-node in-place」更新；不需要 FFN hidden 落地）
 - 注意：為了不增加第三頁 X_mid，本方案允許在 page_out 上做 node-by-node 的 in-place 更新：
 - 對每個 node：
 1) 先把 X_mid(node)=X_out(node) 讀到 reg vec（長度 D_MODEL）
 2) 計算 FFN：y = W2 * ReLU(W1*x + b1) + b2（建議不存 hidden，逐 k 生成並累加）
 3) residual：X_final = X_mid + y
 4) 把 X_final 寫回同一個 node 的 X_out(node)
 - 這種「同一 node 先讀後寫」可保證行為唯一，且不需要額外 scratch buffer。

- S1 對 Debug 策略 D1 的好處（ckpt 可見性）：
 - K/V 是明確落地在 SCRATCH 的固定區段：很適合 CKPT_K_CACHE / CKPT_V_CACHE
 - ATTN_OUT / X_MID / X_OUT 都在 X_out(page_out) 中，可直接就地 halt 後 READ_MEM 讀回
- residual 的強制規則（避免單埠/單 bank 讀寫衝突）：
 - 預設做法：先算 sublayer_out → 寫入 SCR_ATT_OUT（或等價暫存），再做 add：讀 X_in + 讀 SCR_ATT_OUT → 寫 X_out
 - S1(KV Cache) 推薦例外：若 page_in != page_out，且你以「單一 node 的 out_vec[0..D_MODEL-1]」暫存在 reg，再逐 element 讀 X_in 並直接寫 X_out，則可不配置 SCR_ATT_OUT
- LayerNorm 的強制規則（避免隱式 in-place）：
 - 允許「兩 pass」：pass1 只讀 X_in 計算 mean/var；pass2 讀 X_in 並寫 X_out
 - 不允許同一 pass 同時對同一頁做讀寫

【8.4.2) v11：block 內快取（local buffering）強制規則（English: prefetch & reuse）】
- 目標：讓權重讀取變成「burst/prefetch → local reuse」，減少 W bank port 壓力與仲裁 stall。
- 規則：
 1) 任何權重矩陣/向量在 inner-most loop（例如 d_model 迴圈）中，不允許每次都去 W bank 讀 1 個 u32（禁止零碎 random read）。
 2) 必須先以 tile/burst 方式從 W bank 讀出一段（至少對齊 W_LANES），放進 block 的 local regs / 小 buffer，再用多次 MAC 反覆重用。
 3) 對於 Attention/FFN 這種 GEMV/GEMM-like：
  - 建議以輸出向量維度（行(Colume)=d_model 或 d_ffn）做 tile；每次 prefetch 一個 tile 的權重，計算完一批輸出再換下一批。

【8.5) 多 block（pipeline）共用 SRAM：v11.5 採方案1（單一 2-port SRAM + TDM 排程；保留 logical regions）】
你現在的前提是「blocks 級 pipeline」，同時又可能出現「同一頁被多個 block 讀」（例如 residual 的情境）。
v11.5 你選的方案1，目標是：提升 SRAM 利用率（不再被 4 顆 macro 的深度切分卡死），但仍保留 v11 的 deterministic 行為與 TB 可驗證性。

- 核心（English: single physical memory, logical regions preserved）：
  1) SRAM 仍以 4 個 logical regions（X_PAGE0 / X_PAGE1 / SCRATCH / W_REGION）定義位址範圍（由 SramMap.h 的 base/size 決定）。
  2) 物理上使用一顆 true 2-port SRAM（簡寫 SRAM_2P），所有讀寫都打到同一顆 macro；Top 是唯一 owner / arbitrator。
  3) 每 cycle 最多允許 2 個 memory access（每個 port 各 1 次）。超過 2 個的請求必須在 Top 內序列化（stall / retry）。

- Port 行為約束（為了「行為唯一」而不是追求最極致吞吐）：
  - 禁止同 cycle 對同一個 addr_word 做「兩個寫入」。
  - 禁止同 cycle 對同一個 addr_word 做「一讀一寫」（不依賴 macro 的 write-first/read-first 細節；Top 必須主動避開）。
  - 允許同 cycle 對同一個 addr_word 做「兩個讀」（若 macro 不支援，Top 仍可序列化；功能正確優先）。

- Top 的仲裁/排程規則（deterministic arbitration；TB 不會因排程抖動而歧義）：
  - Top 內部把所有 memory access 統一抽象成：{addr_word, is_write, wdata, tag(region, block_id)}。
  - 每個 cycle 依固定優先序挑最多 2 筆派發到 port0/port1；其餘保留到下一 cycle：
    1) 必要寫入（X_out / SCRATCH 的 write-back；避免資料遺失）
    2) 讀 X_in（支撐 datapath）
    3) 讀 W_REGION 的 prefetch（放入各 block 的 local FIFO/line-buffer；見【8.4.2】）
    4) DEBUG/READ_MEM（最低優先；避免干擾推論吞吐）
  - 若你之後要更公平，可在同優先序內採 round-robin，但必須是「固定、可重現」的規則（English: deterministic RR）。

- W_REGION 的 FIFO/prefetch（配合 2-port 才能吃到效益）：
  - 強制精神不變：權重不得在 inner-most loop 逐 word 亂讀；必須先 burst/prefetch 到 local buffer 再重用（【8.4.2】）。
  - 建議綁定：port1 優先服務 W_REGION 的 burst/prefetch；port0 優先服務 X/SCRATCH 的讀寫。
    - 若某 cycle port1 沒有 W prefetch req，允許 port1 借去做 X/SCRATCH access（提高利用率）。
  - 若物理 SRAM 為 32b：W_LANES=8 的寬讀口以「連續 8 次 read」完成，prefetch FIFO 至少要能容納一個 tile（>= 8 words，實務上建議 >= 2~4×W_LANES）。

- Catapult 建模建議（不改外部介面；僅影響內部 SRAM 模型）：
  - 以單一 SRAM array 表示整顆 memory（u32 words），並提供兩個獨立的存取介面/方法（對應 port0/port1）。
  - 或以 Catapult 支援的 dual-port memory pragma / memory configuration 來推導 2-port macro（視你用的 memory compiler/flow）。
  - 無論採哪種建模方式：addr_word 的 decode、range check、以及 READ_MEM 的行為都必須完全依 SramMap.h（保持 TB 一致）。

- 邏輯上的「region lock」仍可保留，但用途改為「資料生命週期/覆寫保護」，而不是假設物理上一定有 4 顆獨立 macro：
  - X_PAGE0/1 的 in_use_mask（per-consumer bitmask）仍然是覆寫保護的唯一依據（見【8.2】）。
  - SCRATCH 的覆寫保證仍由 Top 排程負責（同一時間只允許一個 owner 使用 SCRATCH，或以明確的 alias_group 宣告可重疊）。

【[legacy] v11.4 的 bank/lock 敘述（保留文字；v11.5 方案1 不要求物理 4 顆 macro）】
你現在的前提是「blocks 級 pipeline」，同時又可能出現「同一頁被多個 block 讀」（你提到 residual 的情境）。
因此這裡要把假設寫清楚，否則很容易 TB 可以跑但硬體行為不唯一。

- 方案A 的核心：把 SRAM regions 拆成多個「可獨立鎖定/仲裁」的 bank（至少：X_PAGE0、X_PAGE1、SCRATCH、W；(可選)PERSIST_DEBUG）
-（v11）W bank 內的 W_REGION 必須維持連續位址；禁止用低位址交錯或依權重種類分散到多 bank。
 - 實作層面你有兩種等價做法（擇一即可）：
 1) 物理上真的是多顆 SRAM macro（最直覺，也最容易做 pipeline）
 2) Catapult 以 memory intervals / 多 memory instance 方式建模，讓不同 bank 有各自的 sync/arb（底層仍可能被綜合成 banked memory）

- 互斥/仲裁規則（保證可合成、行為唯一）：
 - 每個 bank 都要有自己的 lock（概念上就是你之前問的 ac_sync/ac_shared）
 - 任一個 block 要讀/寫某個 bank 前，必須先拿到該 bank 的 lock；用完立刻釋放
 - 若兩個 block 同時想讀同一個 bank：
 - 預設行為：其中一個 stall（等待 lock）→ 功能正確，但吞吐可能下降
 - 若你真的要「同 cycle 多讀」：那就必須選擇 multi-port SRAM 或把資料 replicate（這是物理限制）

- residual 導致多讀的處理建議（你說「可能被多個 block 讀」）：
 - 最省事/最穩的做法：把 residual add 留在同一個 TransformerLayer block 內（你現在的 其實就是這樣假設）
 - 這樣「同一個 X_in」的多次讀取都發生在同一個 block 裡，等同單一 requester，不會變成多 block 競爭
 - 若你堅持把 residual add 拆成獨立 block：
 - 就必須接受同一頁會有多 consumer（多 block）讀
 - 需要 refcnt（或 consumer_done token）避免頁被覆寫 + 需要 lock 來序列化同 bank 讀取
 - 上述「refcnt」一律改為 in_use_mask（per-consumer 1-bit bitmask）為主；覆寫條件以 mask==0 判斷；如需數量再用 popcount(mask)

【8.6) Debug/比對（你已選 D1：就地 halt + READ_MEM；預設不需要 PERSIST_DEBUG）】
- D1 基本策略（你選的；最省 SRAM、最「硬體正統」）：
 - DEBUG_CFG 只負責「設定斷點條件」（kind/layer/head + halt_en）
 - INFER 跑到指定 checkpoint 時：
 1) Top 進入 HALTED（freeze）：停止往下運算、停止任何 SRAM 寫入（確保被觀測資料不被覆寫）
 2) 回 ctrl_rsp：RSP_ERR(ERR_DBG_HALT)
 3) 立即在 data_out 送 2 個 u32 meta words：
 - meta0 = addr_word（觀測資料所在 SRAM word address）
 - meta1 = len_words（觀測資料長度，以 u32 words）
 - D1 meta 回應順序固定：先 ctrl_rsp（RSP_ERR(ERR_DBG_HALT)），再 data_out meta0、meta1；Top 需內建 2-word meta buffer，避免 data_out backpressure 影響 ctrl_rsp。

 4) TB 用 READ_MEM(meta0, meta1) 把那段 SRAM 內容讀回做比對/畫圖
 5) TB 需要繼續同一次 INFER：送 DEBUG_CFG(RESUME)；不想繼續則 SOFT_RESET

- 建議的 ckpt(kind) 定義（搭配你已選的 S1：KV Cache）：
 - 1: X_IN （某 layer 進入前的 X_in；在 X_PAGE(page_in)）
 - 2: K_CACHE （SCR_K；shape=[N_NODES,D_MODEL]，列(Row)=node、行(Colume)=d_model）
 - 3: V_CACHE （SCR_V；同上）
 - 4: ATTN_OUT （attention sublayer + residual 後的 X_mid；在 X_PAGE(page_out)）
 - 5: X_OUT （FFN + residual 後的最終 X_out；在 X_PAGE(page_out)）
 - 6: MID_LN_OUT （若啟用 mid LN）
 - 7: END_LN_OUT （end LN 之後）

-（legacy，可保留但你目前不採用）D0：copy-to-PERSIST_DEBUG
 - 若你未來想「不中斷推論、仍保留 checkpoint snapshot」才需要這條路
 - 才需要在 SramMap.h 分配 PERSIST_DEBUG，並在命中點把資料 copy 到固定區域，再由 TB READ_MEM 讀回

【9) bias/base offset 自動化（延伸 ，但符合你新增的「bias 模式」）】
- SRAM base offset/bias 不在硬體中計算，必須離線自動生成（禁止人工手填 base）
- 必須提供離線生成腳本（例如 scripts/gen_sram_map.py），根據：
 - ModelDesc.h / runtime cfg
 - algorithm_ref.ipynb 的命名與 shape
 - weights.h 的存在張量
 自動生成：
 - include/SramMap.h：每個 tensor base offset + size + dtype(FP32/BITPACK) + valid_bits（僅 BITPACK 段需要）
 - include/SramMap.h 必須額外描述（為了方案B 的 ping-pong/scratch 規則，避免「隱式覆寫」）：
 - region：W_REGION / ACT_REGION
 - lifetime_class：PINGPONG / SCRATCH / PERSIST
 - pingpong_tag：若為 PINGPONG，必須標示是 X（以及對應 X_PAGE0/X_PAGE1 的 base）
 - alias/overlay（可選）：若多個 scratch buffer 允許重疊使用，必須以 alias_group 或同 base 明確宣告
 - align_words：各 tensor 的對齊需求（例如 ALIGN_WORDS=16）
 - include/WeightStreamOrder.h：固定傳輸順序 + 每段 expected_len(words) + 段類型
 - include/BiasDesc.h（或 bias_*.h）：bias 的生成結果/expected_len
- Top 在 LOAD_BIAS / LOAD_W：
 - 以 write_ptr 連續寫 SRAM
 - 逐段驗證 expected_len（少送/多送/長度不符 → error）

【10) PreprocEmbedSPE 的 LPE 規格】
- LPE per-node；LPE 佔據 D_MODEL 尾端 slice
 - cfg_d_embed = cfg_d_model - cfg_d_lpe
 - LPE_BASE = cfg_d_embed
 - x[node][LPE_BASE .. cfg_d_model-1] 對應 lpe_token[node][0..cfg_d_lpe-1]
- LPE 不做額外計算：LOAD_W 時載入；PreprocEmbedSPE 只負責把 LPE slice 填入輸出尾端
- PreprocEmbedSPE 必須明確定義輸出 layout（各段起始 offset 與長度）
- 線性化提醒（對應 ModelDesc.h）：
 - lpe_token flatten order = node major, lpe_dim minor
 - idx = node * D_LPE + i
 - node 可視為列(Row)、lpe_dim 可視為行(Colume)

【11) 權重/bit-packed 擺放順序（你要求：依種類分段，同類型跨層連續擺放）】
- 目標：讓 Top「不需要理解權重語意」，只做順序寫入；使用者在 TB 端（weights_streamer.h）自行決定串流順序
- 但順序仍必須遵循「類型分段」規則，且可由 include/WeightStreamOrder.h 驗證：
 A. 校驗矩陣 H（BITPACK；最後一個 word padding=0）
 B. embed（FP32 words）
 C. LPE token（FP32 words）
 D. Mask（若存在，BITPACK）
 E. 量化參數（若存在；例如 s_x / s_w；FP32 words；並包含你列出的 l0/l1 in/o/ff1/ff2 s_x）
 F. Attention weights（每層同類型連續擺放）：
 - Wq (layer0..layerN-1)
 - Wk (layer0..layerN-1)
 - Wv (layer0..layerN-1)
 - Wo (layer0..layerN-1)（若存在）
 G. FFN weights（每層同類型連續擺放）：
 - Wff1 (layer0..)
 - Wff2 (layer0..)
 H. LayerNorm weights（每層同類型連續擺放）：
 - gamma/beta（或等價參數；依 notebook/weights.h 命名）
 I. 其他（FinalHead 相關兩層 FFN、logits head…依 notebook/weights.h）
- 若某類型在不同 layer 的元素數不同：
 - Top 必須有 regs 記錄「每層該類型的 words 數」（例如 layer_wq_words[layer_id]）
 -（v11 固定）上述 regs **只能** 由離線生成的 include/WeightStreamOrder.h 初始化（唯一來源）；CFG 不得再提供第二套 per-layer words 資訊，以避免 TB/Top 行為歧義。
 -（v11 補充）以上所有權重/偏置/bitpack/input y 皆位於 W bank 的連續位址範圍（禁止跨 bank 平均分散/全域 interleave）。
